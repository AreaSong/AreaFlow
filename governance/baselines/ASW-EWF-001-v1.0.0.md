# ASW-EWF-001 企业级项目闭环工作流基线

## 1. 文档定位

本文档是大型软件项目治理与交付的企业级通用基线，覆盖从战略提出、投资决策、需求设计、开发交付、生产运行、效果评估到弃用退役的完整生命周期。

本文档中的“完整”表示：

- 覆盖企业软件项目应具备的主要治理域。
- 每个阶段都有输入、责任、产物、门禁、验证和反馈路径。
- 每一种长期事实都有明确的权威来源。
- 项目能够从目标追溯到代码和生产指标，也能从故障反查到原始决策。
- 功能不仅能够上线，还能够被监控、维护、迁移和安全退役。

本文档不是所有行业的最终合规模板。金融、医疗、政务、军工、硬件、AI 模型等项目必须在本基线上叠加对应行业控制项。

## 2. 引用方式

稳定编号：ASW-EWF-001

当前版本：1.0.0

推荐引用语句：

> 按 ASW-EWF-001 v1.0.0 对当前项目进行企业级闭环治理分析，并输出适用项、缺口、裁剪项、风险和实施顺序。

执行具体功能时可使用：

> 按 ASW-EWF-001 v1.0.0 分析并实施“功能名称”，先完成变更分类、源事实定位、影响分析和门禁判断，再进入代码修改与验证。

如果新版本替代旧版本，应明确写成 ASW-EWF-001 v1.1.0 或更高版本，不应只引用编号而省略版本。

## 3. 规范用语

- 必须：不满足则不能通过对应门禁。
- 应该：默认执行；不执行时必须记录理由。
- 可以：按项目规模、风险和成本裁剪。
- 源事实：某一类事实唯一的权威来源。
- 证据：证明门禁、验收或运行结果成立的可复核材料。
- 所有者：对内容正确性和持续维护最终负责的角色或团队。

## 4. 核心原则

1. 先确认目标与问题，再设计功能。
2. 先确定源事实位置，再创建或修改事实。
3. 设计、代码、测试、文档和生产行为必须一致。
4. 每项工作必须有明确所有者和关闭条件。
5. 每个重要决策必须记录原因、替代方案和后果。
6. 每个高风险变更必须具备灰度、监控、止损和回滚方案。
7. 发布完成不等于需求完成，业务效果达到观察周期后才能关闭。
8. 每项长期能力必须有维护和退出策略。
9. 自动生成优先于人工复制，引用优先于重复描述。
10. 治理强度应与风险、影响面和不可逆性成比例。

## 6. 治理层级

### 6.1 战略层

负责确定企业为什么投入、成功意味着什么，以及何时停止投入。

必须产物：

- 企业愿景或业务战略。
- 可量化的目标和关键结果。
- 预算、期限和风险容忍度。
- 不做事项和退出条件。

### 6.2 项目组合层

负责在多个项目之间进行优先级、资金和人员容量分配。

必须考虑：

- 战略贡献度。
- 收益、成本和机会成本。
- 法规或合同强制性。
- 跨项目依赖。
- 关键人员与平台容量。
- 继续、暂停、合并或终止项目的条件。

### 6.3 项目或项目群层

负责里程碑、范围、预算、风险、依赖和跨团队协调。

必须维护：

- 路线图和里程碑。
- RAID 登记册。
- RACI 责任矩阵。
- 决策日志。
- 依赖清单。
- 状态、风险和预算偏差。

### 6.4 产品与领域层

负责用户问题、业务规则、产品范围、体验和价值验证。

### 6.5 架构与平台层

负责系统边界、技术标准、共享能力、非功能需求和长期演进。

### 6.6 交付团队层

负责需求澄清、实现、测试、文档、发布准备和缺陷修复。

### 6.7 生产运营层

负责可用性、性能、安全运营、客户支持、事故响应、成本和容量。

## 7. 端到端追溯链

~~~mermaid
flowchart LR
    A["OBJ<br/>目标"] --> B["INIT<br/>项目或能力"]
    B --> C["REQ<br/>需求与业务规则"]
    C --> D["RFC<br/>候选方案"]
    D --> E["ADR<br/>最终决策"]
    E --> F["CONTRACT<br/>API、事件、Schema"]
    F --> G["WORK<br/>任务"]
    G --> H["PR<br/>代码变更"]
    H --> I["TEST<br/>验证证据"]
    I --> J["REL<br/>发布记录"]
    J --> K["OBS<br/>生产证据"]
    K --> L["REVIEW<br/>效果复盘"]
    L --> A
~~~

最低追溯要求：

- 需求能够追溯到目标或强制性来源。
- PR 能够追溯到任务、需求和设计决策。
- 测试能够追溯到验收标准和风险。
- 发布能够追溯到具体变更和审批证据。
- 生产指标能够追溯到 SLO 和业务目标。
- 事故改进能够回写到需求、设计、代码、测试或运行手册。

## 8. 生命周期与阶段门禁

| 门禁 | 阶段 | 关键问题 | 必须证据 |
|---|---|---|---|
| G0 | 需求登记 | 问题是否真实存在 | 来源、问题描述、初始所有者 |
| G1 | 立项审批 | 是否值得投入 | 商业论证、成本、风险、优先级 |
| G2 | 需求就绪 | 是否知道要交付什么 | PRD、范围、非目标、验收标准 |
| G3 | 设计就绪 | 是否知道如何安全实现 | RFC/ADR、架构、数据、安全、NFR |
| G4 | 开发就绪 | 是否具备执行条件 | 任务拆分、依赖、环境、负责人 |
| G5 | 合并就绪 | 实现是否符合要求 | 评审、测试、契约、文档、扫描 |
| G6 | 发布就绪 | 是否能安全进入生产 | 灰度、监控、回滚、值班、审批 |
| G7 | 结果关闭 | 是否产生预期价值 | 生产健康、业务指标、观察周期 |
| G8 | 退役关闭 | 是否完整移除能力 | 迁移、数据处置、资源与权限清理 |

任何门禁被豁免时，必须记录：

- 豁免原因。
- 风险接受人。
- 有效期限。
- 补偿控制。
- 后续关闭任务。

## 9. 源事实模型

源事实不是一个巨大文档，而是每一种事实只有一个权威位置。

| 事实类型 | 权威来源 | 不应成为权威来源 |
|---|---|---|
| 战略目标 | Strategy、OKR | 聊天记录、会议口述 |
| 用户问题与产品范围 | PRD | PR 描述、临时任务 |
| 业务术语和规则 | Glossary、Domain Rules | 多份重复 Wiki |
| 候选技术方案 | RFC | 零散讨论消息 |
| 已生效技术决策 | ADR | 仅存在于代码注释 |
| HTTP API | OpenAPI | 手写重复接口说明 |
| RPC 与消息契约 | Proto、AsyncAPI、Schema | 消费方猜测 |
| 数据结构 | Schema、Migration | 截图或人工表格 |
| 字段业务语义 | Data Dictionary | 仅靠字段名推断 |
| 系统实际实现 | 代码与配置 | 过期设计文档 |
| 正确性要求 | 自动化测试、契约测试 | 仅人工口头验收 |
| 工作状态 | Issue Tracker | 周报副本 |
| 生产事实 | 日志、指标、追踪、审计 | 发布前假设 |
| 故障处置 | Runbook | 个人经验 |
| 发布事实 | Release Record、Changelog | 临时通知 |
| 事故结论 | Postmortem | 聊天复盘 |

## 12. 常见功能的强制考虑项

| 功能类型 | 必须考虑 |
|---|---|
| 登录、MFA、SSO | 认证流程、账号恢复、威胁模型、隐私、审计、会话失效 |
| 权限、租户隔离 | 主体、资源、动作、数据范围、默认拒绝、越权测试、审计 |
| 支付、退款、计费 | 状态机、幂等、金额精度、对账、补偿、审计、合规、告警 |
| 订单和审批流 | 状态、事件、并发、撤销、补偿、超时、人工介入 |
| 新增数据库字段 | 字段语义、约束、迁移、兼容期、历史数据、隐私等级 |
| 数据导出或删除 | 身份核验、权限、数据范围、保留、审计、失败恢复 |
| 第三方集成 | 契约、SLA、限流、重试、密钥、数据协议、退出和替代方案 |
| 定时任务 | 时区、幂等、重试、补数、并发、所有者、指标、告警 |
| 消息消费者 | 交付语义、顺序、重复、死信、重放、版本兼容、积压告警 |
| 搜索与报表 | 指标口径、数据来源、权限、时效、准确性、性能 |
| 通知 | 渠道、模板、偏好、退订、重试、去重、隐私、供应商故障 |
| 文件上传 | 类型、大小、病毒扫描、访问控制、存储、保留、删除 |
| 缓存 | 一致性、失效、击穿、穿透、雪崩、容量、降级 |
| 功能开关 | 所有者、默认值、作用域、到期时间、清理任务、失败行为 |
| A/B 实验 | 假设、指标、样本、分组稳定性、隐私、停止条件 |
| 性能优化 | 基线、目标、压测、容量、成本、正确性和回退 |
| Bug 修复 | 复现、根因、影响、回归测试；改变事实时更新对应文档 |
| 重构 | 行为不变证据、测试覆盖、性能影响、回滚和分阶段策略 |
| 依赖升级 | 兼容性、安全公告、许可证、构建与运行验证、回退 |
| 功能退役 | 调用方、用户通知、迁移、数据处置、资源、权限、告警清理 |

## 13. 统一影响分析模型

每项中高风险变更必须逐项判断：

### 13.1 业务

- 解决什么问题。
- 成功指标和当前基线是什么。
- 谁受益、谁承担成本。
- 不做的影响是什么。
- 范围、非目标和退出条件是什么。

### 13.2 用户体验

- 正常、空、加载、失败、重试和撤销状态。
- 多端、时区、语言和可访问性。
- 用户是否理解不可逆操作。
- 旧客户端或旧流程如何兼容。

### 13.3 架构

- 模块和服务边界。
- 同步或异步。
- 单点故障和依赖方向。
- 是否引入长期耦合。
- 是否符合现有架构原则。

### 13.4 数据

- 数据所有者和业务含义。
- Schema、迁移和兼容期。
- 一致性、幂等和并发。
- 保留、归档、删除和审计。
- 数据质量和历史数据处理。

### 13.5 接口与集成

- 契约和错误语义。
- 版本与向后兼容。
- 超时、重试、限流和熔断。
- 事件顺序、重复和重放。
- 第三方不可用时的行为。

### 13.6 安全、隐私与合规

- 身份认证和授权。
- 租户与数据隔离。
- 敏感数据分类和最小化。
- 加密、密钥和审计。
- 滥用场景和威胁模型。
- 法规、合同和数据地域要求。

### 13.7 可靠性与性能

- SLO、延迟、吞吐和容量。
- 超时、重试、降级和恢复。
- 峰值、突发流量和资源上限。
- RTO 与 RPO。
- 成本随规模增长的模型。

### 13.8 测试

- 单元、组件、契约、集成和端到端测试。
- 权限、安全、性能和恢复测试。
- 正常、异常、边界、并发和幂等场景。
- 测试数据、环境和证据保留。

### 13.9 运维

- 日志、指标、追踪和审计。
- 告警阈值、值班和升级路径。
- Runbook、补偿和人工操作。
- 灰度、回滚和发布后观察。
- 客服和业务运营如何识别问题。

### 13.10 组织与成本

- 所有者和协作团队。
- 跨项目依赖和交付时序。
- 人员、云资源、许可证和供应商成本。
- 培训、客服和运营负担。
- 长期维护与退出成本。

## 14. 角色与责任

| 角色 | 最终责任 |
|---|---|
| Executive Sponsor | 战略目标、预算和重大风险接受 |
| Portfolio Owner | 项目优先级和资源组合 |
| Product Owner | 用户价值、范围、业务规则和验收 |
| Project/Program Manager | 里程碑、依赖、风险和跨团队协调 |
| Tech Lead/Architect | 技术方案、架构一致性和技术风险 |
| Engineering Team | 实现、单元验证、技术文档和可维护性 |
| QA/Test Owner | 测试策略、质量证据和发布质量判断 |
| Security/Privacy | 威胁、权限、隐私和合规控制 |
| Data Owner | 数据语义、质量、生命周期和访问规则 |
| SRE/Operations | SLO、可观测性、发布、容量和事故响应 |
| Support/Business Ops | 用户反馈、工单、运营流程和已知问题 |
| Finance/Procurement/Legal | 预算、采购、许可证、合同和供应商风险 |

一个关键事项只能有一个最终责任者，可以有多个执行者和协作者。

## 15. RAID 与依赖治理

RAID 包括 Risk、Assumption、Issue、Dependency。

每条记录必须具有：

- 稳定编号。
- 描述和影响范围。
- 概率、影响和等级。
- 所有者。
- 缓解或解决计划。
- 截止日期。
- 升级条件。
- 当前状态。
- 关闭证据。

假设一旦被证伪，必须转为问题、风险或需求变更，不能继续以假设状态存在。

## 16. 变更分类

| 等级 | 示例 | 最低治理要求 |
|---|---|---|
| L0 | 注释、无行为格式调整 | PR、基础检查 |
| L1 | 局部低风险功能或 Bug | 需求关联、测试、必要文档 |
| L2 | 多模块、API、数据结构变更 | 影响分析、RFC/ADR、集成验证、发布方案 |
| L3 | 权限、支付、迁移、跨服务一致性 | 正式评审、安全与数据审查、灰度、回滚、值班 |
| L4 | 不可逆、监管、重大客户或全局基础设施 | 高层风险接受、演练、业务连续性和正式变更窗口 |

变更等级由影响面、敏感度、可逆性、复杂度和故障后果共同决定。

## 17. Definition of Ready

进入开发前必须满足：

- 问题、目标和范围明确。
- 验收标准可测试。
- 业务术语和规则无关键歧义。
- 依赖、风险和所有者已识别。
- UI 或交互具备必要原型。
- API、数据、安全和合规影响已判断。
- 需要的 RFC 或 ADR 已通过。
- 测试、发布和监控要求已定义。

## 18. Definition of Done

功能完成必须满足：

- 实现符合源事实和编码规范。
- 必要测试通过并保留证据。
- API、事件和数据契约已同步。
- 文档、Runbook 和发布说明已更新。
- 安全、隐私和许可证检查通过。
- 可观测性已部署并验证有效。
- 灰度、回滚和兼容策略已验证。
- 已清理临时代码、过期开关或建立有期限的清理任务。
- 生产观察达到约定周期。
- 业务验收和目标评估完成。

## 19. 发布门禁

发布前必须回答：

1. 发布什么，关联哪些需求、PR 和契约。
2. 谁批准、谁执行、谁值班、谁可决定回滚。
3. 如何灰度，观察哪些技术和业务指标。
4. 什么条件停止扩量。
5. 什么条件必须回滚。
6. 回滚是否包含代码、配置、数据和消息处理。
7. 旧客户端、旧数据和旧调用方是否兼容。
8. 客服、业务和相关团队是否需要通知。

## 20. 环境、配置与密钥

- 环境差异必须显式记录，不能依赖个人记忆。
- 配置必须版本化或由受控配置系统管理。
- 密钥不得进入代码、普通文档、日志或测试样例。
- 配置变更必须具备审计、审批和回滚能力。
- 测试环境不得无控制地复制生产敏感数据。
- 基础设施变更必须经过计划、预览、验证和漂移检查。

## 21. 质量策略

质量门禁应按风险设置，至少包括：

- 格式和静态检查。
- 类型检查和构建。
- 单元测试。
- 契约和集成测试。
- 关键流程端到端测试。
- 依赖和代码安全扫描。
- 数据迁移验证。
- 性能与容量验证。
- 灾难恢复或故障注入演练。

覆盖率只能作为辅助指标，不能替代关键场景、风险和行为验证。

## 22. 安全与隐私

涉及认证、授权、租户隔离、支付、密钥、敏感数据或不可逆操作时，必须升级为高风险流程。

必须明确：

- 信任边界和威胁主体。
- 默认拒绝和最小权限。
- 数据分类、收集目的和保留期限。
- 传输、存储和备份加密。
- 密钥轮换和撤销。
- 审计事件和异常检测。
- 安全事件响应和披露路径。

## 23. 生产运行

每个生产服务必须具有：

- 明确所有者和升级路径。
- 服务目录条目。
- SLI、SLO 和错误预算。
- 日志、指标、追踪和审计。
- 发布、回滚、排障和恢复 Runbook。
- 容量与成本监控。
- 依赖和健康检查。
- 备份、恢复和灾难恢复要求。

## 24. 事故闭环

~~~mermaid
flowchart LR
    A["发现异常"] --> B["分级与响应"]
    B --> C["止损与恢复"]
    C --> D["影响确认与沟通"]
    D --> E["根因分析"]
    E --> F["改进任务"]
    F --> G["代码、测试、架构、Runbook 回写"]
    G --> H["验证改进有效"]
    H --> I["关闭事故"]
~~~

事故复盘应无责但不无行动。每项改进必须有所有者、期限、优先级和验证证据。

## 25. 业务效果闭环

功能上线前必须定义：

- 当前基线。
- 目标指标。
- 反向护栏指标。
- 数据来源和统计口径。
- 观察周期。
- 成功、继续观察、调整或终止的阈值。

不能只观察点击率而忽略错误率、投诉、成本、安全和长期留存。

## 26. 财务、采购与供应商

涉及外部服务或许可证时必须考虑：

- 采购和预算所有者。
- SLA、支持响应和赔偿条件。
- 数据处理和保密协议。
- 数据位置、分包商和合规声明。
- 价格随规模增长的模型。
- 供应商锁定和替代方案。
- 合同终止后的数据导出与删除。
- 供应商故障时的降级或退出方案。

## 27. 弃用与退役

~~~mermaid
flowchart TD
    A["提出退役"] --> B["识别用户、调用方、数据和依赖"]
    B --> C["制定迁移和通知计划"]
    C --> D["停止新增使用"]
    D --> E["迁移存量流量和数据"]
    E --> F["观察无有效使用"]
    F --> G["关闭入口和写入"]
    G --> H["移除代码、契约、配置和资源"]
    H --> I["清理权限、密钥、告警和值班"]
    I --> J["按策略归档或删除数据"]
    J --> K["更新架构、目录、文档和预算"]
    K --> L["证据归档并关闭所有权"]
~~~

功能开关、兼容层、临时表和双写逻辑必须设置所有者和到期时间。

## 28. 文档治理

每份长期文档至少包含：

~~~yaml
id: ADR-0042
title: 订单取消采用异步补偿
status: accepted
owner: team-order
approvers:
  - architecture
  - payment
effective_date: YYYY-MM-DD
last_verified: YYYY-MM-DD
review_cycle: annual
review_triggers:
  - payment-provider-changed
  - order-state-machine-changed
supersedes: ADR-0018
related:
  - REQ-ORDER-108
  - API-order-v3
~~~

文档状态统一为：

~~~mermaid
stateDiagram-v2
    [*] --> Draft
    Draft --> InReview
    InReview --> Draft: 需要修改
    InReview --> Accepted: 评审通过
    Accepted --> Deprecated: 准备停止使用
    Accepted --> Superseded: 被新文档替代
    Deprecated --> Archived: 完成退役
    Superseded --> Archived
~~~

ADR 应通过新 ADR 替代，不能直接删除或重写历史决策。

## 29. 治理节奏

| 频率 | 会议或活动 | 关注点 |
|---|---|---|
| 持续 | CI、监控、告警、安全扫描 | 质量和生产健康 |
| 每日 | 团队同步或运行检查 | 阻塞、风险、事故 |
| 每迭代 | 计划、评审、回顾 | 交付、质量和改进 |
| 每月 | 项目和架构治理 | 里程碑、依赖、风险、技术债、成本 |
| 每季度 | 项目组合和战略复核 | 投资、收益、优先级、暂停或终止 |
| 定期演练 | 恢复、安全和事故演练 | 控制是否真实有效 |

会议不是源事实。会议产生的决策、风险和任务必须写回对应系统。

## 30. 指标体系

### 30.1 价值

- 业务目标完成度。
- 用户采用、留存或任务成功率。
- 收益、成本节约或风险降低。

### 30.2 流动效率

- 从需求到生产的周期。
- 等待时间和返工率。
- 阻塞和跨团队依赖时长。

### 30.3 质量

- 生产缺陷和变更失败率。
- 回滚率和缺陷逃逸率。
- 自动化测试与关键场景覆盖。

### 30.4 可靠性

- SLO 达成率。
- 事故数量、恢复时间和重复事故率。
- 容量、积压和错误预算。

### 30.5 安全

- 高风险发现关闭时间。
- 权限和密钥异常。
- 安全事件和审计覆盖。

### 30.6 成本与可维护性

- 单位业务量成本。
- 云资源与供应商成本。
- 技术债、过期开关、无所有者服务和过期文档。

指标必须用于决策，不能只用于汇报。

## 31. 行业扩展层

在以下场景中必须加载额外治理基线：

- 金融：支付、账务、反洗钱、审计、资金安全和监管报送。
- 医疗：健康数据、患者隐私、临床安全和医疗器械要求。
- 政务：等保、密码应用、数据分级分类和国产化要求。
- AI：数据来源、模型评估、提示注入、偏差、幻觉、人工复核和模型退役。
- 硬件或 IoT：供应链、固件升级、设备身份、离线恢复和物理安全。
- 跨境业务：数据地域、跨境传输、税务、消费者保护和本地法规。

## 32. 裁剪规则

小型项目可以合并角色和文档，但不能删除责任与事实。

允许裁剪的方式：

- RFC 与 ADR 合并为轻量决策记录。
- 项目章程、路线图和 RAID 合并到同一治理文档。
- 小型服务使用简化 Runbook。
- 低风险变更减少审批层级。

不允许裁剪：

- 高风险变更的安全、回滚和验证。
- 生产服务所有者。
- 权限和敏感数据控制。
- 数据迁移和不可逆变更的恢复方案。
- 目标、验收和生产效果闭环。

## 33. 对其他项目的采用流程

~~~mermaid
flowchart TD
    A["读取项目根目录和最近 AGENTS.md"] --> B["识别项目类型、阶段和风险等级"]
    B --> C["盘点代码、文档、基础设施和现有流程"]
    C --> D["按 ASW-EWF-001 建立适用性矩阵"]
    D --> E["标记 已满足、部分满足、缺失、不适用"]
    E --> F["识别高风险缺口和阻塞关系"]
    F --> G["提出最小可行治理补齐计划"]
    G --> H["用户确认高风险或结构性变更"]
    H --> I["分阶段实施"]
    I --> J["运行测试、构建、文档和生产准备验证"]
    J --> K["形成项目自己的治理基线和偏差记录"]
~~~

采用时不得机械复制全部目录和模板。必须先理解项目现状，再决定继承、裁剪和扩展。

## 34. Codex 调用协议

### 34.1 全项目治理检查

> 调用 ASW-EWF-001 v1.0.0。读取当前项目及最近的 AGENTS.md，建立适用性矩阵，按严重程度列出企业级闭环缺口。先只分析，不修改文件。

### 34.2 新功能设计

> 调用 ASW-EWF-001 v1.0.0，对“功能名称”完成需求、架构、数据、API、安全、性能、测试、运维、成本、发布和退役分析，给出源事实文档落点和阶段门禁。

### 34.3 功能实现

> 调用 ASW-EWF-001 v1.0.0 实施“功能名称”。先读取项目上下文，完成变更分类和影响分析；高风险项先请求确认；完成后运行匹配的验证并汇报风险。

### 34.4 发布审查

> 调用 ASW-EWF-001 v1.0.0，对当前发布执行 G6 发布就绪审查，检查变更范围、兼容性、迁移、灰度、监控、回滚、值班和用户通知。

### 34.5 事故复盘

> 调用 ASW-EWF-001 v1.0.0，对该事故建立时间线、影响范围、根因、促成因素和改进任务，并将结论回写到对应源事实。

### 34.6 退役审查

> 调用 ASW-EWF-001 v1.0.0，对“功能或服务名称”执行 G8 退役审查，确认调用方、迁移、数据处置、资源、权限、密钥、告警、预算和文档均已关闭。

## 35. 完整性检查清单

- 战略目标、预算和退出条件是否明确。
- 项目优先级和资源是否经过组合治理。
- 需求是否有来源、所有者、范围和验收标准。
- 设计是否覆盖架构、数据、接口、安全和非功能需求。
- 是否有 RACI、RAID、依赖和决策记录。
- 代码、测试、契约和文档是否一致。
- 发布是否有灰度、监控、停止条件和回滚。
- 生产服务是否有所有者、SLO、Runbook 和灾备。
- 事故是否能形成有验证证据的改进任务。
- 业务效果是否回流到下一轮投资和需求决策。
- 财务、采购、法律和供应商风险是否受控。
- 功能是否具备弃用、迁移和安全退役能力。
- 每类长期事实是否只有一个权威来源。
- 文档是否有所有者、状态、版本和复审触发条件。
- 行业专项要求是否通过扩展层补充。

## 36. 版本治理

版本规则：

- Patch：文字澄清、示例修正，不改变治理要求。
- Minor：新增兼容的治理模块、门禁或模板。
- Major：改变阶段、责任、强制要求或引用方式。

变更本基线时必须记录：

- 修改原因。
- 受影响章节。
- 与旧版本的兼容性。
- 采用方需要执行的迁移动作。

## 37. 最终判定

符合 ASW-EWF-001 的项目，应当能够证明：

> 每一个长期事实有唯一权威来源；每一个目标能够追溯到生产结果；每一个决策能够解释其原因和后果；每一个交付都经过与风险匹配的验证；每一个生产问题都能回流并形成改进；每一项能力最终都能够被安全退役。
