# v0.8 Multi-project / Multi-worker

## Goal

AreaFlow 进入多项目、多 worker 平台阶段。
Worker scheduling 的长期合同见
[`../architecture/worker-scheduling-contract.md`](../architecture/worker-scheduling-contract.md)；本 milestone
只打开 summary / schedule preview / readiness，不打开真实自动 scheduler。v0.8 阶段合同见
[`../architecture/v0.8-multi-project-worker-pool-contract.md`](../architecture/v0.8-multi-project-worker-pool-contract.md)。

## Includes

- 多项目注册。
- Worker pool。
- Read-only worker pool summary:
  `GET /api/v1/worker-pool/summary` and `areaflow worker pool-summary`。
- Read-only worker pool schedule preview:
  `GET /api/v1/worker-pool/schedule-preview` and `areaflow worker schedule-preview`。
- Project priority。
- Agent roles。
- Resource limits。
- Engine profile readiness。
- Concurrency controls。

## Excludes

- 不引入第二套状态源事实。
- 不允许 worker 跨 project scope 越权。
- 不打开真实自动 scheduler、远程 worker credential、secret resolve、team/auth enforcement 或多项目
  execution apply。

## Success

- 多个 project 可同时注册和查询。
- Worker pool summary 能跨项目显示 worker、online worker、active lease、queued task、
  needs_recovery 和 capability 集合，且不触发调度、lease recovery 或 worker command。
- Schedule preview 能基于 summary 输出 recommended / blocked、blocked_reasons、available_slots
  和 next_action，且 `policy.dry_run_only = true`。
- Project scheduling policy 能从 `areaflow.yaml` 进入 PostgreSQL，并在 worker pool summary 与
  schedule preview 中展示 priority、max_parallel_tasks、agent_role、required_capabilities 和
  engine_profile。
- Schedule preview 能按 project scope、priority、agent role、required capabilities 和 resource readiness
  解释调度建议，但不创建 lease、不启动 worker。
- 真实 scheduler、idempotent schedule command、team/auth enforcement 和 remote worker credential 保持关闭，
  等后续独立 gate 打开。
- Engine profile readiness 不泄露 secret，不跨项目复用未授权能力，也不调用真实 engine routing apply。

## v0.8a Boundary

v0.8a 只打开 worker pool 的观测层。它复用 PostgreSQL 作为状态源事实，读取 `projects`、
`workers`、`leases` 和 `run_tasks` 的聚合状态。

v0.8a 不做：

- 自动调度。
- 自动领取 lease。
- 自动 recovery。
- 跨项目 worker 执行。
- resource limit enforcement。

后续 priority、resource limit、agent role 和 engine routing apply 都必须基于这个只读 summary 再增加
显式 command / scheduler 能力。

## v0.8b Boundary

v0.8b 打开 worker pool 的只读 schedule preview。默认策略：

```text
strategy = default_fifo
default_project_priority = 100
slot_strategy = online_workers_minus_active_leases
dry_run_only = true
```

preview 可以把 project 分成：

- `recommended=true`：存在 queued task、online worker、available slot，且 worker capability 包含
  `read_project`。
- `recommended=false`：返回 `blocked_reasons`，例如 `no_queued_tasks`、`no_online_workers`、
  `no_available_worker_slots` 或 `missing_read_project_capability`。

v0.8b 不做真实 scheduler，不创建 lease，不调用 worker `run-once`，不写 event/audit。
Preview 不能复用真实 acquire lease path；它只能读取聚合状态并解释 `recommended`、`blocked_reasons`、
`available_slots` 和 `next_action`。

## v0.8c Boundary

v0.8c 将项目级调度策略接入只读 preview：

```text
priority
max_parallel_tasks
agent_role
required_capabilities
engine_profile
```

策略来源是项目配置，落入 `project_scheduling_policies`。`worker pool summary` 返回每个项目的
策略；`schedule preview` 使用项目 priority 排序，用 `max_parallel_tasks` 限制可用 slot，并逐项检查
`required_capabilities`。

slot 计算为：

```text
worker_slots = online_workers - active_leases
project_slots = max_parallel_tasks - active_leases
available_slots = min(worker_slots, project_slots)
```

下限为 0。缺少 required capability 时返回：

```text
missing_required_capability:<capability>
```

`max_parallel_tasks` 是 project-scoped 并发上限。AreaMatrix dogfood 的第一阶段真实 execution 必须保持
`max_parallel_tasks=1`；即使 preview 显示未来可用 slot，也不能被解释为真实 AreaMatrix 并发执行已经打开。

v0.8c 仍然只读，不做：

- 自动调度。
- 自动领取 lease。
- 调用 worker `run-once`。
- 写 event/audit。
- 强制 resource limit。
- 校验 engine secret 或打开真实 engine routing apply。

## v0.8d Boundary

v0.8d 增加 engine/resource readiness preview。它读取项目配置中的：

```text
engines.default
engines.profiles[].id
engines.profiles[].provider
engines.profiles[].enabled
engines.profiles[].secret_ref
engines.profiles[].resource_limits
```

AreaFlow 将所选 `engine_profile` 的只读 readiness 写入 `project_scheduling_policies.metadata`，
并在 worker pool summary 与 schedule preview 中返回：

```text
engine.profile_id
engine.provider
engine.enabled
engine.secret_ref
engine.secret_required
engine.secret_ready
engine.resource_limits
engine.status
engine.blocked_reasons
```

当前不会解析 secret 明文。`secret_ref = none` 视为不需要 secret；其他 `secret_ref` 在本阶段都只标记为
`secret_ref_unavailable`，等待 secret store 设计落地。`enabled: false` 会让 preview 返回
`engine_profile_disabled`，但不会写 event/audit，因为 preview 仍是只读观察面。

v0.8d 不做：

- 调用 Codex CLI、OpenAI 或任何 AI engine。
- 解析 OS keychain/env secret。
- 打开真实 engine routing apply。
- 强制 resource limit。
- 自动调度或创建 lease。

## v0.8e Boundary

v0.8e 增加 resource limit readiness preview。它只解释 engine profile 中的已知资源限制：

```text
resource_limits.max_active_leases
resource_limits.max_queued_tasks
```

AreaFlow 在 worker pool summary 与 schedule preview 中返回：

```text
resources.max_active_leases
resources.max_queued_tasks
resources.status
resources.blocked_reasons
```

当 `active_leases >= max_active_leases` 时，preview 返回：

```text
resource_limit:max_active_leases
```

当 `queued_tasks > max_queued_tasks` 时，preview 返回：

```text
resource_limit:max_queued_tasks
```

v0.8e 仍是只读 readiness，不做：

- 资源扣减。
- 强制调度限流。
- 自动 lease recovery。
- 调用 worker 或 engine。
- 写 event/audit。

## v0.8f Boundary

v0.8f 增加 agent role readiness preview。`worker pool summary` 会聚合 online worker 的
`worker_type`，`schedule preview` 会用项目策略中的 `agent_role` 检查是否有可用 worker 类型。

当前匹配规则：

```text
worker_type == agent_role
local_host satisfies local_worker
```

如果没有匹配 worker，preview 返回：

```text
missing_agent_role:<agent_role>
```

API/CLI/Web 返回：

```text
worker_types
role.required_role
role.matched
role.matched_types
role.status
role.blocked_reasons
```

v0.8f 不做：

- 真实 worker 选择。
- 跨项目 worker 分配。
- agent role 自动升级或 fallback。
- 创建 lease 或调用 `run-once`。
- 写 event/audit。

## v0.8g Boundary

v0.8g 增加 PostgreSQL `project_key` isolation fixture。它创建两个临时 project，并使用相同
workflow version label 验证 Store 查询、worker、lease、schedule preview 和 lease recovery 都必须按
project scope 隔离。

Smoke 入口：

```bash
bash scripts/smoke-project-isolation.sh
make smoke-project-isolation
make smoke-docker-project-isolation
```

覆盖范围：

```text
workflow_versions
runs
artifacts
events
audit_events
workers
leases / lease recovery
```

v0.8g 不做：

- 真实 scheduler。
- 远程 worker。
- secret / API token / team scope。
- 跨项目 worker 执行。
- 写被管理项目文件。

## v0.8h Boundary

v0.8h 增加 API route scope contract test。它使用两个 project 和相同 `shared-v1` workflow label，
验证 versioned API route 必须把 `{project_key}` 解析为 service layer 的 project scope。

覆盖范围：

```text
GET /api/v1/projects/{project_key}/summary
GET /api/v1/projects/{project_key}/events
GET /api/v1/audit-events?project_key=<key>
GET /api/v1/projects/{project_key}/workers
GET /api/v1/projects/{project_key}/workflow-versions
GET /api/v1/projects/{project_key}/workflow-versions/{label}/runs
GET /api/v1/projects/{project_key}/artifacts
GET /api/v1/projects/{project_key}/workflow-versions/{label}/artifacts
```

验证命令：

```bash
go test ./internal/api -run TestProjectScopedAPIsUseRouteProjectKey -count=1
```

v0.8h 不做：

- API token / team / user 权限隔离。
- 真实多项目调度。
- 远程 worker credential scope。

## v0.8i Boundary

v0.8i 增加 global run/artifact ID route 的兼容型 visibility guard。全局 ID route 继续支持本机
single-user 兼容调用；当请求提供 `project_key` 时，API 必须先把该 key 解析成 project scope，并确认
run/artifact metadata 的 `project_id` 与 scope 匹配。

覆盖范围：

```text
GET /api/v1/runs/{run_id}?project_key=<key>
GET /api/v1/runs/{run_id}/events?project_key=<key>
GET /api/v1/runs/{run_id}/events/stream?project_key=<key>
GET /api/v1/runs/{run_id}/execution-approval-gate?project_key=<key>
POST /api/v1/runs/{run_id}/start?project_key=<key>
GET /api/v1/artifacts/{artifact_id}?project_key=<key>
GET /api/v1/artifacts/{artifact_id}/content?project_key=<key>
```

匹配 project scope 时返回正常响应；不匹配时返回 `404`，避免通过全局 ID route 探测其他 project 的
run/artifact 是否存在。

验证命令：

```bash
go test ./internal/api -run 'TestGlobalRunEndpointsHonorProjectKeyVisibility|TestGlobalArtifactEndpointsHonorProjectKeyVisibility' -count=1
```

v0.8i 不做：

- 强制所有本机兼容调用都必须传 `project_key`。
- API token / team / user 权限隔离。
- secret scope。
- 真实多项目调度。
- 远程 worker credential scope。

Auth、team、API token、secret resolve 和 remote worker credential 的 R4 ladder 见
[`../architecture/auth-team-secret-boundary.md`](../architecture/auth-team-secret-boundary.md)。v0.8 只记录边界和
readiness，不启用真实 enforcement。
