## v0.9a Local Service Status

先提供 Desktop shell 可复用的只读服务状态契约，而不是直接实现完整 Tauri 壳。

```text
GET /api/v1/service/status
GET /api/v1/desktop/service-control-gate
GET /api/v1/desktop/notification-gate
GET /api/v1/desktop/tray-menu-gate
areaflow service status --json
areaflow desktop service-control-gate --json
areaflow desktop notification-gate --json
areaflow desktop tray-menu-gate --json
```

该状态返回：

- API、PostgreSQL、worker pool 和 dashboard 入口状态。
- worker pool 的 total projects、workers、online workers、active leases、queued tasks 和 recovery 数量。
- Desktop/Web/CLI 可观察能力：`observe_api`、`observe_database`、`observe_worker_pool`、
  `open_web_dashboard`、`stream_events`。
- 明确禁止动作：`write_project_files`、`run_workflow_directly`、`maintain_second_database`、
  `bypass_areaflow_api`。

## Includes

- Tauri shell。
- Local service status display。
- Desktop service-control gate display。
- Desktop notification gate display。
- Desktop tray/menu gate display。
- Dashboard launcher。
- Worker status。

## Constraint

Desktop 不重新实现业务逻辑，只调用 AreaFlow API。
Desktop 只把 `service status` 当成观察面和 launcher 输入；任何状态变更仍必须经过 AreaFlow
API/service layer、gate、capability 和 audit。
Desktop service-control gate 只展示 start/stop/restart、notification 和 tray/menu 的禁用原因；
在 process supervision、drain/restart recovery、notification permission、audit 和 rollback contract
完成前不得直接执行本机控制动作。
Desktop notification gate 只展示 SSE / OS notification 的只读要求；在 event filter、redaction、
dedupe、rate limit、OS permission 和 audit contract 完成前不得请求系统通知权限或建立通知桥。
Desktop tray-menu gate 只展示菜单项门禁；在 OS integration、service control、notification、settings
和 audit contract 完成前不得创建原生 tray/menu 或从菜单动作执行写操作。
v0.9 Desktop 只面向本机 local service，不承担团队远程控制台职责；远程团队/多用户控制台放到 v1.x
后续阶段，并按 [`../architecture/team-remote-control-boundary.md`](../architecture/team-remote-control-boundary.md)
逐级打开。
Desktop 也不能把 service status 当成 process control，不能触发 support bundle export、远程 telemetry、
managed upgrade 或 rollback apply。

## Excludes

- Desktop 不直接改项目文件。
- Desktop 不维护第二数据库。
- Desktop 不直接执行 workflow。
- v0.9a 不实现完整 Tauri shell、系统通知、secret UI 或 worker 调度。
- v0.9 不实现真实 process start/stop/restart、OS notification permission request、native tray/menu、
  远程团队控制台、多用户管理面或绕过 API 的 worker 控制。

## Success

- `GET /api/v1/service/status` 可返回 local service 只读状态。
- `GET /api/v1/desktop/service-control-gate` 可返回 service control 只读门禁。
- `GET /api/v1/desktop/notification-gate` 可返回 notification / SSE 只读门禁。
- `GET /api/v1/desktop/tray-menu-gate` 可返回 menu bar / tray 只读门禁。
- `areaflow service status --json` 可返回同一语义的 CLI JSON。
- `areaflow desktop service-control-gate --json`、`areaflow desktop notification-gate --json` 和
  `areaflow desktop tray-menu-gate --json` 可返回与 API/Desktop shell 相同的只读 gate 语义。
- smoke 覆盖 service status、dashboard URL、API URL、worker pool totals、capabilities 和 forbidden actions。
- Desktop 能观察 local service，并展示 start/stop/restart 为什么启用或禁用。
- Desktop 能显示 PG/API/worker 健康状态。
- Desktop 能打开 Web dashboard。
- 系统通知路径只能来自 AreaFlow API/SSE；notification gate 通过前不建立 native notification bridge。
- 系统通知在 notification gate 通过前保持 disabled，不请求 OS permission。
- 菜单栏/托盘在 tray-menu gate 通过前保持 disabled，不创建 native tray。
- 本机 secret 来源不写入项目配置。
