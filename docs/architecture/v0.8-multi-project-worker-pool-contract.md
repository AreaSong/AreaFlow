# v0.8 Multi-project Worker Pool Contract

## Purpose

本文定义 AreaFlow v0.8 `Multi-project Worker Pool` 的阶段边界。它补充
[`worker-scheduling-contract.md`](worker-scheduling-contract.md)、
[`api-surface.md`](api-surface.md)、
[`data-model-v1.md`](data-model-v1.md)、
[`v0.7-web-dashboard-contract.md`](v0.7-web-dashboard-contract.md)、
[`auth-team-secret-boundary.md`](auth-team-secret-boundary.md) 和
[`../milestones/v0.8-multi-project-worker.md`](../milestones/v0.8-multi-project-worker.md)。

v0.8 的目标是把 AreaFlow 从 AreaMatrix 单项目 dogfood 扩展到多项目可观察 worker pool：项目级 worker、
lease、run_task、scheduling policy、engine readiness、resource readiness 和 route isolation 都能稳定查询
和预览。v0.8 不打开真实自动 scheduler、远程 worker、secret resolve、team permission enforcement 或跨项目
execution apply。

核心规则：v0.8 的 `summary`、`schedule preview`、engine/resource/role readiness 和 project isolation 证据
只能证明多项目状态可解释、可隔离、可观察；不能证明真实 scheduler、远程 worker、secret-backed engine 或
多项目自动执行已经打开。

## Position In Roadmap

v0.8 位于 Web Dashboard 之后、Desktop Shell 之前：

```text
Web Dashboard
-> Multi-project Worker Pool
-> Desktop Shell
-> Stable Platform
```

v0.8 只允许推进：

```text
multi-project registration visibility
worker pool summary
schedule preview
project scheduling policy display
project-scoped slot calculation
agent role readiness
engine profile readiness
resource limit readiness
project_key store isolation fixture
project-scoped API route tests
global run/artifact ID visibility guard
Web display of worker pool preview
```

v0.8 不允许推进：

```text
automatic scheduler loop
schedule.acquire / schedule.dispatch command
cross-project worker execution
remote worker credential issuance
secret resolve
API token enforcement
team permission enforcement
engine routing apply
worker auto-run from preview
lease creation from schedule preview
event/audit writes from preview
resource quota enforcement
AreaMatrix parallel execution > 1
```

## Query Surface

v0.8 Query API surfaces are read-only:

```text
GET /api/v1/worker-pool/summary
GET /api/v1/worker-pool/schedule-preview
GET /api/v1/projects/{project_key}/workers
GET /api/v1/projects/{project_key}/workflow-versions
GET /api/v1/projects/{project_key}/workflow-versions/{version}/runs
GET /api/v1/projects/{project_key}/artifacts
GET /api/v1/projects/{project_key}/workflow-versions/{version}/artifacts
GET /api/v1/runs/{run_id}?project_key={project_key}
GET /api/v1/artifacts/{artifact_id}?project_key={project_key}
```

The corresponding CLI surfaces are also read-only:

```text
areaflow worker pool-summary
areaflow worker schedule-preview
```

These surfaces must not create command requests, leases, run attempts, artifacts, events, audit events, worker
commands, engine calls or managed project writes.

## Worker Pool Summary Boundary

`worker-pool/summary` may aggregate:

```text
projects
workers
worker_heartbeats
leases
run_tasks
project_scheduling_policies
engine profile metadata
resource limit metadata
```

It may return:

```text
workers / online_workers / offline_workers
active_leases / needs_recovery_leases
queued_tasks / needs_recovery_tasks
capabilities
worker_types
scheduling.priority
scheduling.max_parallel_tasks
scheduling.agent_role
scheduling.required_capabilities
scheduling.engine_profile
role readiness
engine readiness
resource readiness
```

It must not:

```text
recover expired leases
change worker status
claim tasks
start run-once
write event/audit
resolve secret
call engine
write project files
```

## Schedule Preview Boundary

`worker-pool/schedule-preview` is an explanation layer over summary. It may compute:

```text
recommended
blocked
blocked_reasons
available_slots
next_action
policy.dry_run_only=true
```

`recommended=true` means only:

```text
there is at least one queued task
there is an online matching worker
slot calculation is positive
required capability, role, engine and resource readiness do not block preview
```

It does not mean a task is approved, a lease was created, a worker will run, or AreaMatrix execution can proceed.
`next_action=worker_run_once_preview` is a preview label, not an instruction to call `worker run-once`.

Preview must never reuse the real acquire lease path. If future scheduler code shares helper functions, the helper
must be pure/read-only and must not take a transaction lock intended for mutation.

## Project Scope Boundary

`project_key` / `project_id` is the isolation boundary for:

```text
workflow_versions
runs
run_tasks
run_attempts
artifacts
workers
leases
events
audit_events
secret_refs
engine_profiles
```

Versioned API routes must resolve the route `{project_key}` first and pass that project scope to the service layer.
Global run/artifact ID routes may keep local single-user compatibility without `project_key`, but when `project_key`
is supplied they must enforce visibility guard:

```text
matching project scope:
  return the normal response.

mismatched project scope:
  return 404 without revealing whether the global ID exists elsewhere.
```

This guard is not a full auth model. API token, user, team, secret and remote worker scopes remain closed until the
R4 ladder in [`auth-team-secret-boundary.md`](auth-team-secret-boundary.md) is opened.

## Scheduling Policy Boundary

Project scheduling policy may come from `areaflow.yaml` and be persisted as `project_scheduling_policies`:

```text
priority
max_parallel_tasks
agent_role
required_capabilities
engine_profile
metadata
```

v0.8 may show and use these fields for preview ordering and slot calculation. It must not use them to bypass approval,
permission, execution gates or AreaMatrix protected paths.

AreaMatrix dogfood keeps:

```text
max_parallel_tasks=1
real execution concurrency=1
```

even if preview displays higher theoretical capacity for other fixture projects. Any increase for real AreaMatrix
execution requires a separate gate, explicit approval and evidence.

## Engine And Secret Readiness Boundary

v0.8 may read engine profile metadata:

```text
profile_id
provider
enabled
secret_ref
resource_limits
```

It may return readiness:

```text
secret_ref=none -> secret_required=false, secret_ready=true
secret_ref=<name> -> secret_required=true, secret_ready=false, blocked_reasons includes secret_ref_unavailable
enabled=false -> blocked_reasons includes engine_profile_disabled
```

It must not:

```text
read env vars
read OS keychain
read encrypted DB secret plaintext
call Codex CLI / OpenAI / local model
inject secret into worker
test provider credentials
```

`secret_ready=true` in v0.8 can only mean no secret is required for the preview path. It cannot prove secret-backed
execution readiness.

## Role And Resource Readiness Boundary

v0.8 agent role matching is preview-only:

```text
worker_type == agent_role
local_host satisfies local_worker
```

Missing role returns:

```text
missing_agent_role:<role>
```

Resource limit readiness may interpret:

```text
resource_limits.max_active_leases
resource_limits.max_queued_tasks
```

It may return blockers such as:

```text
resource_limit:max_active_leases
resource_limit:max_queued_tasks
```

These blockers explain readiness only. v0.8 does not decrement quotas, reserve capacity, enforce scheduler throttling,
cancel runs or recover leases.

## Remote Worker Boundary

`remote_worker` is a label/readiness blocker in v0.8, not an execution capability.

v0.8 must not:

```text
issue remote worker credentials
rotate or revoke remote worker tokens
allow remote worker direct PostgreSQL access
dispatch tasks to remote workers
upload remote artifacts
expose lease-scoped credentials
```

Remote worker credential issuance belongs to the R4 ladder in
[`auth-team-secret-boundary.md`](auth-team-secret-boundary.md). Until then, `remote_worker` should appear only as
`missing_agent_role`, `blocked`, `preview_only` or explicit readiness metadata.

## Web And Desktop Boundary

Web v0.7 may display v0.8 summary and schedule preview. Desktop v0.9 may later display the same fields through local
service status. Neither surface may call scheduler apply, start workers or create leases from preview.

UI wording must preserve preview semantics:

```text
recommended = suggested candidate for a future protected command
blocked = why scheduler is not ready
next_action = preview label, not executable command
available_slots = computed capacity, not reserved capacity
```

## Handoff To v0.9

v0.8 handoff to v0.9 requires evidence that:

- Worker pool summary is read-only and project-scoped.
- Schedule preview returns `policy.dry_run_only=true`.
- Schedule preview does not create leases, events or audit rows.
- Project scheduling policy fields appear in summary/preview.
- Engine readiness does not resolve secrets.
- Agent role and resource blockers are visible.
- Store project isolation fixture compiles and has a safe skip path without PostgreSQL.
- API project route scope and global ID visibility guard tests pass.
- Web displays worker pool preview without non-GET `/api/v1` requests.
- AreaMatrix protected paths remain untouched.

v0.9 can then show these facts in Desktop service status. v0.9 must not treat v0.8 preview as scheduler apply.

## Smoke Evidence

v0.8 evidence is split by boundary:

```text
worker pool summary / schedule preview:
  go test ./internal/api -run 'TestWorkerPool|TestWorkerPoolEndpointsUseAPIV1' -count=1
  go run ./cmd/areaflow worker pool-summary --json
  go run ./cmd/areaflow worker schedule-preview --json

project route isolation:
  go test ./internal/api -run 'TestProjectScopedAPIsUseRouteProjectKey|TestGlobalRunEndpointsHonorProjectKeyVisibility|TestGlobalArtifactEndpointsHonorProjectKeyVisibility' -count=1

PostgreSQL store isolation:
  AREAFLOW_DATABASE_URL=postgres://... make smoke-project-isolation
  make smoke-docker-project-isolation

Web observation:
  scripts/smoke-web.sh
```

If PostgreSQL is not available, `TestStoreProjectKeyIsolationWithPostgres` may skip safely; that proves the fixture is
safe to run, not that PostgreSQL isolation passed. Multi-project completion claims require a real PostgreSQL pass
recorded in evidence.

## Anti-patterns

v0.8 must reject these interpretations:

- “recommended=true, so the scheduler is running.”
- “available_slots > 0, so capacity is reserved.”
- “remote_worker appears in policy, so remote worker credentials exist.”
- “secret_ref is configured, so engine secret can be resolved.”
- “route guard passed, so team/user auth is complete.”
- “worker pool summary can recover leases.”
- “schedule preview can call worker run-once.”
- “AreaMatrix can increase concurrency because preview slots exist.”
