# v0.6 Worker Execution Beta

阶段合同见 [`../architecture/v0.6-worker-beta-contract.md`](../contracts/v0.6-worker-beta-contract.md)。
本阶段只打开 worker lifecycle、lease、dry-run worker loop 和有限 scoped execution evidence；不打开
真实 AreaMatrix execution cutover、`./task-loop run` forwarding、真实 engine 调用、source write、
checkpoint apply 或 repair apply。

## Goal

开始执行已批准任务。

v0.6 的 execution beta 受
[`../architecture/execution-opening-strategy.md`](../../../../proposals/execution-opening.md) 约束。
本文每个 boundary 只能打开自己列出的能力；不能把 fixture、read-only verify、artifact-only 或 rollback
drill 解释成真实 AreaMatrix execution cutover。

## Includes

- Codex CLI adapter。
- Worker mode。
- Lease / heartbeat。
- Copy -> verify -> repair。
- Logs / artifacts / audit events。

## Constraint

不重写 AreaMatrix v1 历史。

## Success

- Worker 能通过 lease 获取 approved run_task。
- Lease / heartbeat 防止同一 run_task 被重复执行。
- Worker 崩溃后任务进入 needs_recovery，而不是直接丢失或误判完成。
- Copy -> verify -> repair 保持 attempt 历史。
- 只有 verify PASS 后 workflow item 才能进入 done。
- Checkpoint fail 会阻断下一任务。
- Drain 能跑完当前 task 的 verify/checkpoint 后停止。
- 兼容状态 `verified`、`artifact_written`、`rollback_verified` 只能解释为 scoped outcome；目标模型收敛到
  `status=passed` + outcome / attempt / artifact / gate evidence。

## Completion Semantics

v0.6 的每条 boundary 只证明自己列出的 scope：

```text
fixture execution:
  证明 PG state / artifact store execution apply 闭环。

read-only verify:
  证明 allowlisted target path 的 hash/size evidence 可生成。

approved artifact write:
  证明 AreaFlow-owned artifact store 写入可审计。

fixture project write:
  证明 fixture 单文件 write/verify/rollback 可恢复。

fixture/temp managed generated write:
  证明 fixture/temp generated path write/verify/rollback 可恢复。

generated write readiness / beta gate:
  证明进入人工审查或 approval gate，不打开 apply。
```

这些证据不能累计成真实 AreaMatrix execution cutover。真实 cutover 还必须证明 approved task 的 copy、
verify、repair/checkpoint gate、artifact/evidence、audit、compat command behavior、protected-path proof 和
rollback/remediation 全部可复验。

## v0.6a Boundary

v0.6a 先落地 worker registry、heartbeat 和 lease schema：

- Migration `000006_v0_6_worker_registry.sql` 增加 `workers`、`worker_heartbeats` 和 `leases`。
- `areaflow worker register <project>` 注册或更新 worker，并创建 worker actor。
- `areaflow worker heartbeat <project> <worker-key>` 记录心跳。
- `areaflow worker list <project>` 查询项目 worker。
- REST API 提供 `GET/POST /api/v1/projects/{project_key}/workers`。
- REST API 提供 `POST /api/v1/projects/{project_key}/workers/{worker_key}/heartbeat`。
- register / heartbeat 写入 `events` 和 `audit_events`。
- register / heartbeat 完成 `command_requests` response，并支持显式 `idempotency_key` replay / conflict。
- worker lifecycle command response 必须记录 `project_write_attempted=false`、`execution_write_attempted=false`、
  `engine_call_attempted=false`、`commands_run=false`、`secrets_resolved=false`、`network_used=false`、
  `lease_created=false`、`attempt_created=false`、`artifact_created=false` 和 `worker_run_once=false`。

v0.6a 不领取任务、不执行 copy/verify、不调用 Codex CLI、不写被管理项目文件；它只证明 worker registry
和 heartbeat 的 AreaFlow 内部状态写入具备 command 幂等边界。

## v0.6b Boundary

v0.6b 增加最小 lease lifecycle：

- `areaflow worker lease-acquire <project> <worker-key> --run-task-id <id>` 领取已有 dry-run `run_task`。
- `areaflow worker lease-release <project> <worker-key> --lease-id <id>` 释放 active lease。
- `areaflow worker lease-recover <project>` 将过期 active lease 标记为 `needs_recovery`。
- REST API 提供 `POST /api/v1/projects/{project_key}/workers/{worker_key}/lease-acquire`。
- REST API 提供 `POST /api/v1/projects/{project_key}/workers/{worker_key}/lease-release`。
- REST API 提供 `POST /api/v1/projects/{project_key}/workers/lease-recover`。
- 同一 `run_task` 同时只能存在一个 active lease，由 PostgreSQL partial unique index 兜底。
- acquire / release / recover 写入 `events` 和 `audit_events`。
- acquire / release / recover 完成 `command_requests` response。
- lease command response 必须记录 `project_write_attempted=false`、`execution_write_attempted=false`、
  `engine_call_attempted=false`、`commands_run=false`、`secrets_resolved=false`、`network_used=false`、
  `attempt_created=false`、`artifact_created=false` 和 `worker_run_once=false`。
- capability denied 路径必须返回 `decision=denied`、`lease_created=false`，并且不创建 lease、attempt
  或 artifact。

v0.6b 仍不执行任务、不调用 Codex CLI、不写被管理项目文件；它只证明 worker 能安全领取和释放
AreaFlow 数据库中的执行单元。

## v0.6c Boundary

v0.6c 增加 worker dry-run `run-once`：

- `areaflow worker run-once <project> <worker-key>` 自动寻找一个已有 dry-run `run_task`。
- run-once 会 recovery 过期 lease、领取一个 active lease、记录 dry-run 结果、释放 lease。
- REST API 提供 `POST /api/v1/projects/{project_key}/workers/{worker_key}/run-once`。
- 没有可领取任务时返回 `claimed=false`，不报错。
- 成功领取时返回 `worker`、`lease`、`task` 和 `claimed=true`。
- run-once 写入 `events` 和 `audit_events`。

v0.6c 仍不执行 copy/verify，不调用 Codex CLI，不写被管理项目文件；它只证明 worker loop 能在
AreaFlow 数据库中安全跑一拍。

## v0.6d Boundary

v0.6d 给 worker dry-run `run-once` 补齐 evidence：

- 成功领取时创建 `run_attempts.attempt_kind = worker_run_once`。
- attempt 为 `dry_run=true`、`status=passed`。
- 写入 local artifact store 的 `worker_run_once_report`。
- PostgreSQL 只保存 artifact metadata、URI、sha256、size 和 content type。
- CLI / API JSON 返回 `attempt` 和 `artifact`。
- lease release metadata、event 和 audit event 带上 `attempt_id` 与 `artifact_id`。

v0.6d 仍不执行 copy/verify，不调用 Codex CLI，不写被管理项目文件；它只证明 worker loop 能在
AreaFlow 数据库和 artifact store 中留下可审计 evidence。

## v0.6e Boundary

v0.6e 给 worker lease / run-once 增加 capability preflight：

- `lease-acquire` 请求的 `allowed_capabilities` 必须是 worker 注册 `capabilities` 的子集。
- `run-once` 请求的 `allowed_capabilities` 必须是 worker 注册 `capabilities` 的子集。
- worker 缺少请求 capability 时，操作返回 capability denied，不创建 lease、attempt 或 artifact。
- REST API 返回 `403 worker capability denied`。
- denied 路径写入 `events` 和 `audit_events`，metadata 包含 requested、worker 和 missing capabilities。

v0.6e 仍不执行 copy/verify，不调用 Codex CLI，不写被管理项目文件；它只证明 worker 不能通过请求参数扩大自己的 capability。

## v0.6f Boundary

v0.6f 给 worker dry-run `run-once` 增加可选 run scope：

- CLI 支持 `areaflow worker run-once <project> <worker-key> --run-id <id>`。
- REST API request 支持 `run_id`。
- 未传 `run_id` 时保持项目级队列行为，领取任意 eligible dry-run `run_task`。
- 传入 `run_id` 时只领取该 run 下的 eligible dry-run `run_task`。
- 没有匹配任务时返回 `claimed=false`，并写入带 `run_id` 的 idle event。
- capability denied、lease scope、report artifact metadata 和 audit metadata 都保留 run scope。

v0.6f 仍不执行 copy/verify，不调用 Codex CLI，不写被管理项目文件；它只让 CLI/API/Web 后续能定向驱动某一次 run。

- preview 返回 engine/profile readiness、proposed command、capability preflight、path preflight 和

- preview 显式返回 `project_write_attempted=false`、`execution_write_attempted=false`、

- 默认 AreaMatrix config 应 blocked 于 disabled engine profile、`run_commands=false` 和

v0.6g 不执行真实 Codex CLI、不解析 secret、不运行 shell、不写 artifact store、不写被管理项目文件；
它只证明 execution beta 打开前的 engine / command / permission / redaction 边界可以被统一预览。

## v0.6h Boundary

- gate 返回 `pass | blocked`、items、blockers、required capabilities、engine preview、worker 列表和 safety facts。
- dry-run runner preview run 必须 blocked，不能被误当作真实 execution apply 候选。
- gate 显式返回 `project_write_attempted=false`、`execution_write_attempted=false`、
  `engine_call_attempted=false`、`commands_run=false`、`secrets_resolved=false`、`network_used=false`、
  `task_claimed=false`、`worker_started=false`、`attempt_created=false` 和 `artifact_created=false`。

v0.6h 不创建 `command_requests`，不领取 task，不启动 worker，不创建 attempt/artifact，不调用 Codex CLI，
不解析 secret，不运行 shell，不写被管理项目文件；它只证明真实 execution apply 之前的 go/no-go 门禁
可以被 CLI/API/Web/Desktop 统一读取。

## v0.6i Boundary

v0.6i 增加 approval-gated fixture execution apply：

- `areaflow run fixture-queue <project> <version>` 创建非 dry-run execution run 和一个
  `fixture_execution_task`。
- `areaflow worker fixture-execute <project> <worker-key> --run-id <id>` 在 execution approval gate
  通过后领取 fixture task。
- REST API 提供 `POST /api/v1/projects/{project_key}/workflow-versions/{version}/fixture-queue`。
- REST API 提供 `POST /api/v1/projects/{project_key}/workers/{worker_key}/fixture-execute`。
- apply 成功时创建 `leases.lease_kind = fixture_execution`、`run_attempts.attempt_kind = fixture_execution`
  和 `fixture_execution_report` artifact。
- apply 成功时将 `run_task.status` 更新为 `passed`，并在无剩余 fixture task 时将 `run.status`
  更新为 `passed`。
- `run.fixture_queue` 和 `worker.fixture_execute` 都写入 completed `command_requests` response，
  并支持显式 idempotency key replay / conflict。
- response 必须记录 `project_write_attempted=false`、`execution_write_attempted=false`、
  `engine_call_attempted=false`、`commands_run=false`、`secrets_resolved=false` 和
  `network_used=false`。

v0.6i 只允许 fixture-only execution apply。它不执行 copy/verify/repair，不调用 Codex CLI，
不运行 shell，不解析 secret，不访问网络，不写真实 AreaMatrix 或其他被管理项目文件，也不写
`workflow/versions/**/execution/**`。它只证明 approval-gated execution apply 能在 AreaFlow 自身
PG state 和 artifact store 中闭环。

## v0.6j Boundary

v0.6j 增加 approval-gated read-only verify：

- `areaflow run read-only-verify-queue <project> <version> --target-path <path>` 创建非 dry-run
  execution run 和一个 `read_only_verify_task`。
- `areaflow worker read-only-verify <project> <worker-key> --run-id <id>` 在 execution approval gate
  通过后领取 read-only verify task。
- REST API 提供 `POST /api/v1/projects/{project_key}/workflow-versions/{version}/read-only-verify-queue`。
- REST API 提供 `POST /api/v1/projects/{project_key}/workers/{worker_key}/read-only-verify`。
- verify 成功时先检查 worker capability，再检查 project permission allowlist 和 project-root 安全路径。
- verify 只读取 allowlisted target file，计算 sha256 和 size；不把 target file 原文写入 artifact。
- verify 成功时创建 `leases.lease_kind = read_only_verify`、`run_attempts.attempt_kind = read_only_verify`
  和 `read_only_verify_report` artifact。

v0.6j 只允许 approval-gated read-only verify。它不执行 copy/verify/repair，不调用 Codex CLI，
不运行 shell，不解析 secret，不访问网络，不写真实 AreaMatrix 或其他被管理项目文件，也不写
`workflow/versions/**/execution/**`。它只证明 worker 可以在 gate 和 permission 保护下读取被管理
项目 allowlisted 文件，并把 hash/size evidence 写入 AreaFlow 自身 PG state 和 artifact store。

## v0.6m Boundary

v0.6m 增加 approved project write design gate。该 gate 只关闭真实项目写入前的协议和门禁，不执行 apply：

- `areaflow run project-write-design-gate <run-id>` 查询只读设计门禁。
- REST API 提供 `GET /api/v1/runs/{run_id}/project-write-design-gate`。
- 定义 `approved_project_write` 的 write-set contract。
- 定义 future copy attempt 的输入 artifact、preimage artifact、post-write hash evidence 和 audit 事实。
- 定义 verify attempt 不可省略，copy success 不等于 done。
- 定义 repair 必须先生成 failure summary 和 repair plan，不能直接改项目。
- 定义 rollback 只能对 AreaFlow 自己 apply 的 write-set 做受控反向写入。
- 定义 checkpoint 必须在 verify pass 后单独 gate，checkpoint fail 会阻断下一 task。
- 明确 first apply 必须先走 fixture approved project write，不直接写真实 AreaMatrix。
- 明确第一版不支持 delete、move、chmod、binary rewrite、symlink target、project-root 外路径或 glob 批量写入。

- `areaflow run fixture-project-write-queue <project> <version> --target-path <path> --content <text>
  --expected-before-sha256 <hash> --expected-before-size <size>` 创建非 dry-run execution run、一个
  `fixture_project_write_task` 和 `fixture_project_write_set` artifact。
- `areaflow worker fixture-project-write <project> <worker-key> --run-id <id>` 在 execution approval gate

- REST API 提供 `POST /api/v1/projects/{project_key}/workflow-versions/{version}/fixture-project-write-queue`。

## v0.6o Boundary

不执行 apply：

- `areaflow run managed-generated-write-gate <run-id>` 查询 generated-only write 门禁。
- REST API 提供 `GET /api/v1/runs/{run_id}/managed-generated-write-gate`。
- gate 返回 `allowed_generated_prefixes`，默认包含 `.areaflow/generated/` 和 `.areamatrix/generated/`。
- gate 返回 `required_write_set_fields`，必须包含 `generated_only`、expected-before hash/size、

  checkpoint、repair 和 destructive operations blocked。
- gate 返回 `apply_sequence`，必须保持 fixture write/verify/rollback drill 在 generated-only apply 之前。

## v0.6q Boundary

- `areaflow project generated-write-readiness <project>` 查询项目级 generated write readiness。
- REST API 提供 `GET /api/v1/projects/{project_key}/generated-write-readiness`。

  AreaMatrix 文件。
- readiness 检查 `read_project`、`write_artifacts` 和 `write_generated` capability 是否在 config 与
  permission rows 中同时允许。
- readiness 检查 `.areaflow/generated/**` 和 `.areamatrix/generated/**` 是否进入 `write_paths`，且有
  `write_generated` path allow，并且未被 forbidden paths deny。

  capability 关闭状态。
- readiness 返回 `ready_for_review=true|false`，用于说明是否具备人工审查资格。

## v0.6r Boundary
