# v1.0 Stable Platform

## Goal

AreaFlow 成为稳定平台。
v1.0 总阶段合同见
[`../architecture/v1.0-stable-platform-contract.md`](../contracts/v1.0-stable-platform-contract.md)。该合同定义
release final gate、completion audit、AreaMatrix dogfood completion、protected path proof 和 v1.x
high-risk apply closed 边界。

## Includes

- Migration compatibility。
- Backup / restore。
- Security review。
- Security boundary readiness。
- Audit completeness。
- Release packaging。
- Public docs。
- Adapter/profile stable boundary。
- Template/profile marketplace seed。

Artifact、backup、restore、archive 和 retention 的长期边界见
[`../architecture/artifact-backup-restore-contract.md`](../contracts/artifact-backup-restore-contract.md)。
Object backend、archive copy/upload、retention-aware GC 和 delete apply 的长期边界见
[`../architecture/object-artifact-retention-contract.md`](../contracts/object-artifact-retention-contract.md)。
v1.0 只交付只读 manifest / integrity / restore dry-run / metadata-only archive preview gate；真实 restore
apply、archive copy/upload、object storage upload、GC/delete 和 orphan cleanup 仍是 v1.x R4。

Release readiness、exception、final gate、evidence、package/distribution/publish/rollout preview 和
真实 publish apply 的长期边界见
[`../architecture/release-final-gate-contract.md`](../contracts/release-final-gate-contract.md)。v1.0
release final gate 是只读 go/no-go，不是 package、migration、exception record、tag、push、sign、
upload、publish 或 restore apply 授权。
0-100% 最终完成证明以
[`../architecture/completion-audit-contract.md`](../contracts/completion-audit-contract.md) 为准；release
final gate `pass` 仍不能单独声明 AreaFlow 已达 100%。

Plugin / marketplace seed、manifest draft、conformance 和未来 plugin execution 的长期边界见
[`../architecture/plugin-marketplace-boundary.md`](../../../../proposals/plugin-marketplace.md)。v1.0
只允许 built-in / seed catalog 和 template/profile metadata，不安装、启用、执行或远程拉取未知 plugin。
Operations、deployment、observability、support bundle、telemetry、upgrade 和 rollback 的长期边界见
[`../architecture/operations-deployment-observability-boundary.md`](../contracts/operations-deployment-observability-boundary.md)。
v1.0 只允许本机 bootstrap/readiness/doctor/service status、metadata-only support bundle preview 和
local-only telemetry；远程运维控制、托管升级、破坏性 rollback 和完整 support export 是 v1.x。

## Hard Boundary

v1.0 是稳定平台边界，不是高风险自动化边界。v1.0 必须证明状态可恢复、权限可解释、证据可审计、
发布可预演、adapter/profile contract 可验证；但以下能力仍保持关闭，进入 v1.x 后再单独设计：

- 真实 restore apply。
- archive copy/upload、object storage upload、retention-aware GC、orphan cleanup 或 delete apply。
- 真实 secret 解析和 engine secret 注入。
- API token enforcement、team permission enforcement 和 remote worker credential issuance。
- 未知第三方 plugin 代码执行。
- 未知第三方 plugin install / enable / remote fetch。
- Webhook delivery、inbound callback processing、external API connector 或 provider notification。
- 自动 tag、push、upload、sign、publish。
- release exception record 真实写入，除非 migration approval gate、apply preview 和 R4 approval 全部通过。
- permission、adapter/profile、backup、local artifact integrity、secret、project isolation 或 protected
  path proof 失败通过 release exception 放行。
- 远程多用户控制台、远程 worker 和团队权限操作面。
- 真实 budget / quota enforcement、rate-limit decrement、usage charge 或 provider billing sync。
- 远程运维控制、自动升级、破坏性 rollback、完整 support bundle export 或默认远程 telemetry。

Auth、team、API token、secret resolve 和 remote worker credential 必须按
[`../architecture/auth-team-secret-boundary.md`](../../../../proposals/auth-team-secret.md) 的 R4 ladder
逐级打开；v1.0 只允许 schema/readiness/doctor/preview 证据。
Team Console 和远程控制台必须按
[`../architecture/team-remote-control-boundary.md`](../../../../proposals/team-and-remote-control.md) 逐级打开；
v1.0 只允许 role matrix preview、readiness 和 blocked reason。
Budget、quota、rate limit 和 usage metering 必须按
[`../architecture/budget-quota-boundary.md`](../../../../proposals/budget-and-quota.md) 逐级打开；v1.0 只允许
estimate / readiness / blocked reason，不允许真实 quota decrement、charge、provider billing sync 或
silent throttling。
External integrations、webhooks、third-party callbacks 和多 API 接入必须按
[`../architecture/integration-webhook-boundary.md`](../../../../proposals/integrations-and-webhooks.md) 逐级打开；
v1.0 只允许 catalog/readiness/delivery plan preview/blocked reason，不投递 webhook、不接受 callback
作为业务事实、不调用外部 API。

## v1.0 Ops / Deployment / Observability Readiness

提供本机可诊断的运维边界，用来证明 AreaFlow 不是只能在开发机临时跑通。

第一版覆盖项：

- install / migrate / server start / project register smoke。
- health、readiness、doctor 和 service status 分层。
- migration ledger 记录 preflight、apply、verify 和 remediation。
- support bundle metadata-only preview。
- logs / diagnostics redaction policy。
- telemetry 默认 local-only。
- Desktop service-control / notification / tray-menu gate 保持只读。

当前只读入口：

```text
GET /api/v1/ops/readiness
GET /api/v1/ops/support-bundle-preview
GET /api/v1/ops/migration-ledger-readiness
areaflow ops readiness --json
areaflow ops migration-ledger-readiness --json
areaflow support bundle-preview --json
```

该 readiness 不执行 auto upgrade、不做 destructive rollback、不打开 remote ops control、不上传 telemetry、
不导出 raw support bundle、不读取 secret、不复制 prompt / artifact / user file 原文，也不修改 AreaMatrix
protected paths。

## v1.0a Backup Manifest

先落地只读 backup manifest，用来证明 PostgreSQL 状态和 artifact metadata 可被完整枚举。

```text
GET /api/v1/backup/manifest
areaflow backup manifest --json
```

该 manifest 包含：

- PostgreSQL 关键表的 row count。
- 每个 project 的基本信息、import inventory 和 artifact metadata。
- `manifest_hash`，用于校验同一份 metadata 清单的稳定形状。
- backup 当前允许能力：`export_postgres_metadata`、`export_artifact_metadata`、`verify_manifest_hash`。
- backup 当前禁止动作：`read_artifact_contents`、`write_project_files`、`restore_database`、
  `delete_existing_state`、`resolve_secrets`。

v1.0a 不读取 artifact 原文、不生成压缩包、不做 restore、不删除或覆盖任何状态。真正 restore 属于后续
高风险能力，必须另行定义影响、权限、验证和回滚。

## v1.0b Audit Coverage

提供只读 audit coverage matrix，用来检查当前 `audit_events` 是否覆盖 v1.0 要求的关键动作。

```text
GET /api/v1/audit/coverage
GET /api/v1/audit/coverage?project_key=areamatrix
areaflow audit coverage --project areamatrix --json
```

第一版覆盖项：

- project registration / config upsert。
- status mirror write。
- workflow authoring。
- approval decision。
- runner preview。
- worker registration。
- worker capability denial。
- worker lease lifecycle。
- real command execution。
- secret resolution。
- permission change。

当前未启用的长期能力必须显示为 `gap`，不能假装完成。尤其是真实命令执行、secret 解析和
permission change 在能力打开前都应作为 audit gap 保留。

## v1.0c Permission Policy Doctor

提供只读 permission policy doctor，用来检查项目权限策略是否仍满足默认只读、显式 allow、deny 优先和
审计可解释性要求。

```text
GET /api/v1/permissions/doctor?project_key=areamatrix
areaflow permissions doctor areamatrix --json
```

第一版覆盖项：

- active project config 是否存在。
- 高风险 capability 是否默认关闭。
- status export 是否只允许写 `.areaflow/status.json`。
- `workflow/versions/*/execution/**`、`progress.json`、app metadata 和本地 DB 文件是否有 deny。
- `./task-loop run`、`git reset --hard`、`git checkout --`、`rm -rf` 是否有命令 deny。
- secret、network 和 git capability 启用时是否返回 `warn`。
- worker scheduling 是否声明 `read_project` 和 `write_artifacts`。
- doctor 本身保持只读，不写 `audit_events`。

该 doctor 不修改 `areaflow.yaml`，不写被管理项目，不执行命令，不读取 secret。AreaMatrix baseline
应返回 `pass`；长期高风险能力一旦打开，必须返回 `warn` 或 `fail` 并要求额外 gate / approval /
audit 证据。

## v1.0d Artifact Integrity Check

提供只读 artifact integrity check，用来证明 PostgreSQL artifact metadata 与 AreaFlow-owned local
artifact 原文一致，并诚实暴露历史 project reference 原文仍留在被管理项目中的事实。

```text
GET /api/v1/artifacts/integrity?project_key=areamatrix
areaflow artifact integrity areamatrix --json
```

第一版覆盖项：

- `local` artifact 必须存在、可读，且 sha256 / size 与 PG metadata 一致。
- `external_project` / `project_reference` artifact 只检查 metadata 形状，不读取被管理项目原文。
- `object` artifact 在本机 doctor 中返回 `skipped` / `needs_attention`，等待对象存储 verifier；不能计入
  完整可恢复内容。
- 未知 backend 返回 `warn`。
- 缺失、不可读、hash mismatch 或 size mismatch 返回 `fail`。

该检查不修复 artifact，不删除文件，不写项目文件，不写 `audit_events`，也不读取 secret。当前 AreaMatrix
baseline 同时包含 AreaFlow-owned local artifact 和历史 `external_project` 引用，因此顶层 status 应为
`warn`，而不是把 skipped 历史引用伪装成完整 `pass`。

## v1.0e Restore Dry-run Plan

提供只读 restore dry-run plan，把 backup manifest、project inventory、artifact inventory 和 artifact
integrity 串成恢复前检查链。

```text
GET /api/v1/backup/restore-plan
areaflow backup restore-plan --json
```

第一版覆盖项：

- manifest shape 是否包含 schema version 和 manifest hash。
- project inventory 是否可枚举。
- artifact inventory 是否包含 local artifact 和历史 project reference artifact。
- dry-run guardrail 是否明确禁止 `restore_database`、`write_project_files`、`delete_existing_state`、
  `resolve_secrets` 和 `apply_restore`。
- 每个 project 的 artifact integrity 状态是否会阻断或需要人工处理。

该计划不读取外部 backup 包，不写数据库，不写 artifact store，不覆盖项目文件，不读取 secret，也不执行
restore apply。当前 AreaMatrix baseline 因历史 `external_project` 引用存在，应返回 `needs_attention`，
用于提示 cutover/archive 前需要决定是否复制历史原文到 AreaFlow artifact store 或对象存储。
如果存在 object artifact，只有 object verifier 已证明 key/version、sha256、size、namespace、encryption、
access policy 和 retention policy 后，restore dry-run 才能把它计入完整可恢复内容。

真实 restore apply 不属于 v1.0 默认交付范围。它放到 v1.x 后续阶段，必须重新定义输入包格式、影响范围、
权限、R4 approval、校验命令、回滚策略和 audit evidence；不能由 restore dry-run 或 release readiness
隐式启用。

## v1.0f Adapter/Profile Conformance

提供只读 adapter/profile conformance check，用来证明 AreaMatrix adapter、AreaMatrix workflow profile
和 AreaFlow core 的边界稳定。
Plugin / marketplace seed 边界见
[`../architecture/plugin-marketplace-boundary.md`](../../../../proposals/plugin-marketplace.md)；本阶段
conformance 不执行第三方 plugin 代码。

```text
GET /api/v1/conformance?project_key=areamatrix
areaflow conformance check areamatrix --json
```

第一版覆盖项：

- project `adapter` / `workflow_profile` 必须匹配内置 profile defaults。
- 内置 profile 必须能加载，并生成稳定 sha256。
- profile validate 必须通过，warning 不隐藏。
- AreaMatrix profile 必须保留固定 item state 枚举。
- AreaMatrix profile 必须保留 16 个固定 stage 顺序。
- AreaMatrix profile 必须保留 17 个固定 gate 名称。
- AreaMatrix profile 必须保留固定 transition 顺序和 required gate，防止 workflow lifecycle 语义漂移。
- AreaMatrix profile 必须保留防止 apply、execution、cutover 和 closeout 被误开的 hard rules。
- AreaMatrix profile 必须保留 PG metadata、artifact store content、managed project source docs 和
  AreaFlow generated output 的 artifact ownership/storage policy。
- AreaMatrix profile 必须保留 authoring cutover 与 execution cutover 的分层策略。
- AreaMatrix adapter 必须能只读加载 snapshot inventory、residual、artifact 和 status hash。
- adapter/profile/core 边界必须证明 metadata-only、只读、不执行命令、不解析 secret、不写数据库。
- active `areaflow.yaml` snapshot 必须证明当前 AreaMatrix 安全基线仍关闭 workflow/code/command/worker/git/
  network/secret/agent 等高风险 capability。

该检查不写项目文件，不写数据库，不写 `audit_events`，不执行命令，不读取 secret，也不启动 worker。
当前 AreaMatrix baseline 应返回 `pass`。

## v1.0g Release Readiness

提供只读 release readiness report，把 v1.0 平台门禁聚合成一个发布前判断。

```text
GET /api/v1/release/readiness
areaflow release readiness --json
```

第一版聚合：

- backup manifest。
- restore dry-run plan。
- audit coverage。
- 每个 project 的 permission policy doctor。
- 每个 project 的 artifact integrity。
- 每个 project 的 adapter/profile conformance。

状态规则：

- `ready`：所有聚合项都 ready/pass。
- `needs_attention`：存在 warn、skipped、needs_attention 或需要人工接受的长期 gap。
- `blocked`：存在 fail、blocked 或缺失关键恢复/权限/边界证据。

该检查不创建发布包，不写数据库，不写项目文件，不写 artifact store，不执行 restore，不执行 cutover，
不解析 secret，也不启动 worker。当前 AreaMatrix baseline 因 restore dry-run 仍有历史 project reference
artifact、audit coverage 仍有未启用长期能力 gap、artifact integrity 仍有 skipped 引用，因此应返回
`needs_attention`，不能伪装成 `ready`。

## v1.0i Release Acceptance Preview

提供只读 release acceptance preview，用来预演 remediation action 是否可以进入显式 release exception

## v1.0j Release Acceptance Gate

提供只读 release acceptance gate，用来把 acceptance preview 转换成发布前 gate 判断。

```text
GET /api/v1/release/acceptance-gate
areaflow release acceptance-gate --json
```

第一版输出：

该 gate 不写数据库，不写项目文件，不写 artifact store，不自动接受 gap，不创建 approval，不执行命令，
不启动 worker，也不 apply release。当前 AreaMatrix baseline 应返回 `blocked`，因为 restore、
audit 和 artifact 三类 exception 仍缺显式接受证据。

## v1.0k Release Exception Doctor

提供只读 release exception doctor，用来在启用真实 exception record 写入前检查记录字段、审计动作和
写入 guardrail。

```text
GET /api/v1/release/exception-doctor
areaflow release exception-doctor --json
```

第一版检查：

- exception record 必需字段：`exception_key`、`source_decision`、`acceptance_type`、`owner`、
  `reason`、`required_evidence`、`expires_or_review_at`、`rollback_plan`、`audit_event_id`。
- exception audit contract 必需动作：`release.exception.request`、`release.exception.approve`、
  `release.exception.revoke`。
- doctor 自身仍保持只读，不写数据库、不 mark accepted、不创建 approval、不 apply release。
- acceptance gate 中 `needs_decision` 项必须生成对应 exception requirement。
- `not_acceptable` 项必须返回 `fail`，不能被 exception doctor 转成可接受。

状态规则：

- `pass`：所有 gate item ready，且 exception 写入能力已明确启用并通过检查。
- `warn`：exception 写入仍关闭，或存在可接受但未决的 exception requirement。
- `fail`：存在不可接受 blocker、未知 decision 状态、或 gate/preview 缺失关键范围。

当前 baseline 应返回 `warn`：schema 和 audit contract 已作为必需项暴露，但真实 exception 写入仍关闭。

## v1.0n Release Exception Migration Approval Gate

提供只读 release exception migration approval gate，用来阻止真实 `release_exceptions` migration
在缺少显式 approval 时被创建或执行。

```text
GET /api/v1/release/exception-migration-approval-gate
areaflow release exception-migration-approval-gate --json
```
