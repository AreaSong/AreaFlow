# v0.3 Version Authoring Contract

## Purpose

本文定义 AreaFlow v0.3 `New Version Authoring` 的阶段边界。它补充
[`workflow-engine-contract.md`](../../../architecture/workflow-engine-contract.md)、
[`adapter-profile-boundary.md`](../../../architecture/adapter-profile-boundary.md)、
[`command-approval-contract.md`](./command-approval-contract.md)、
[`api-surface.md`](./api-surface.md)、
[`v0.2-shadow-doctor-contract.md`](./v0.2-shadow-doctor-contract.md)、
[`../dogfood/areamatrix-contract.md`](../../../dogfood/areamatrix-contract.md) 和
[`../migration/areamatrix-workflow-migration.md`](../migrations/areamatrix-workflow-migration.md)。

v0.3 的目标是让 AreaFlow 可以在自己的 PostgreSQL 和 artifact store 中创建、查询和审计新的
workflow version authoring 候选。它建立 version、stage skeleton、workflow item trace、gate result、
transition preview 和 approval record 的平台语义。

核心规则：v0.3 authoring 是 AreaFlow-owned planning state，不是 AreaMatrix authoring cutover，也不是
execution cutover。任何 version create、gate pass、transition preview ready 或 approval approved 都不能被解释为
promotion apply、被管理项目写入、task-loop 替代、worker execution 或 AI engine 调用已经打开。

## Position In Migration

v0.3 位于 shadow 和 authoring cutover 之间：

```text
Import + Status Mirror
-> Shadow Doctor + Drift Check
-> New Version Authoring
-> Workflow Ownership Cutover
```

进入 v0.3 前必须已经满足：

```text
v0.1 import/mirror metadata baseline exists
v0.2 doctor / summary / readiness / import-diff / verify-bundle are queryable
native doctor authorization boundary is explicit
AreaMatrix protected paths are clean, or changes are separately authorized mirror writes
```

v0.3 完成后只能进入 v0.4 cutover readiness 和 authoring cutover apply 设计。v0.3 本身不得切换
AreaMatrix 的 authoring source of truth。

## Minimal Closed Loop

v0.3 最小闭环是：

```text
workflow version create
-> workflow version list/show
-> workflow version stages
-> workflow gate run
-> workflow transition preview
-> workflow approval record
-> workflow gate list / transition list / approval list
-> protected path proof
```

fixture smoke 可以使用 `workflow version mark-ready` 生成 non-placeholder marker artifact，用于证明 gate /
transition / approval 链的正路径。`mark-ready` 是 AreaFlow-owned authoring evidence，不是被管理项目写入。

## Required CLI Commands

v0.3 必须稳定支持：

```text
areaflow workflow version create <project_key> <label> [--json]
areaflow workflow version list <project_key> [--json]
areaflow workflow version show <project_key> <label> [--json]
areaflow workflow version stages <project_key> <label> [--json]
areaflow workflow version ensure-skeleton <project_key> <label> [--json]
areaflow workflow version mark-ready <project_key> <label> --stage <stage> --item-type <type> [--json]
areaflow workflow gate run <project_key> <label> <gate_name> [--json]
areaflow workflow gate list <project_key> <label> [--json]
areaflow workflow transition preview <project_key> <label> [--json]
areaflow workflow transition list <project_key> <label> [--json]
areaflow workflow approval record <project_key> <label> [--json]
areaflow workflow approval list <project_key> <label> [--json]
```

`mark-ready` must remain an authoring / fixture helper. It can write AreaFlow-owned marker artifacts but cannot write
project files, execution packages, checkpoints, logs, source code or status projection.

```text
GET /api/v1/projects/{project_key}/workflow-versions
GET /api/v1/projects/{project_key}/workflow-versions/{label}
GET /api/v1/projects/{project_key}/workflow-versions/{label}/stages

GET /api/v1/projects/{project_key}/workflow-versions/{label}/transition-previews
GET /api/v1/projects/{project_key}/workflow-versions/{label}/approvals
```

v0.3 Command API:

```text
POST /api/v1/projects/{project_key}/workflow-versions
POST /api/v1/projects/{project_key}/workflow-versions/{label}/stages
POST /api/v1/projects/{project_key}/workflow-versions/{label}/gates
POST /api/v1/projects/{project_key}/workflow-versions/{label}/transition-previews
POST /api/v1/projects/{project_key}/workflow-versions/{label}/approvals
```

These command routes may write AreaFlow PostgreSQL, AreaFlow-owned artifact store, events, audit events and
command requests. They must not write managed project files or start execution.

## Slices

v0.3 is split into four slices:

```text
v0.3a:
  DB-only workflow version create/list/show and profile binding freeze.

v0.3b:
  AreaFlow-owned stage skeleton, workflow items, item links and local artifact placeholders.

v0.3c:
  Gate result model for discussion_gate, plan_doctor, draft_doctor, queue_doctor and promotion_preview.

v0.3d:
  Transition preview and approval record model.
```

These slices can be implemented incrementally, but their safety boundary is shared: no AreaMatrix workflow directory
write, no promotion apply, no execution package, no worker, no engine, no secret resolve.

## Workflow Version Requirements

An AreaFlow-authored workflow version must store:

```text
project_id
display_label
version_kind
lifecycle_status = draft
import_mode = authored
immutable = false
status_summary.profile_binding.profile_id
status_summary.profile_binding.profile_version
status_summary.profile_binding.profile_hash
status_summary.profile_binding.profile_path
```

Label validation must reject empty labels, labels that do not start with an alphanumeric character, labels over the
current limit, path-like labels and labels containing spaces or shell-special characters.

`workflow.version.create` must be idempotent:

```text
same idempotency_key + same request hash -> return existing result
same idempotency_key + different request hash -> reject
same display_label in same project -> reject unless idempotent replay
same display_label in another project -> allowed
```

AreaMatrix import can refresh metadata-only imported versions, but it must not delete or overwrite
`import_mode=authored` versions.

## Stage Skeleton Requirements

v0.3 creates the authoring skeleton from `discussion` through `promotion_preview`:

```text
discussion
middle_layer
changes
plans
drafts
queue
promotion_preview
```

The skeleton must include separate workflow items for:

```text
discussion_package
middle_layer_ledger
change_ledger
plan
draft_manifest
draft_copy
draft_verify
queue_candidate
promotion_preview
```

`draft_copy` and `draft_verify` must remain separate workflow items and separate artifact types. Combining them hides
the implementation / verification split and is not allowed.

The current v0.3 skeleton intentionally does not create full 16-stage lifecycle items for:

```text
intake
source_docs
templates
execution
run
projection
closeout
```

Those stages can be imported, referenced, previewed or added in later phases, but their absence from the v0.3
authoring skeleton is not a failure.

## Artifact Requirements

v0.3 skeleton artifacts are AreaFlow-owned local artifacts:

```text
storage_backend = local
uri points to AreaFlow artifact store
sha256 exists
size_bytes exists
metadata.owned_by = areaflow
```

Artifact files must be written under AreaFlow-owned artifact storage, not under AreaMatrix or any managed project root.
Historical AreaMatrix artifacts remain `external_project` / `project_reference` metadata until a later archive/copy gate.

Placeholder artifacts must be clearly marked as placeholders. A placeholder skeleton can satisfy structural existence,
but it cannot make a content gate pass.

## Workflow Item Link Requirements

v0.3 must preserve traceability through `workflow_item_links`:

```text
discussion -> middle_layer
middle_layer -> changes
changes -> plans
plans -> draft_manifest
draft_manifest -> draft_copy
draft_manifest -> draft_verify
draft_copy -> queue
draft_verify -> queue
queue -> promotion_preview
```

Links must be project-scoped and version-scoped. They cannot cross `project_key`, cannot link imported history into an
authored version without explicit import/reference metadata, and cannot imply promotion or execution.

## Gate Requirements

v0.3 gates are repeatable, append-only checks. They write `gate_results`, events and audit events.

Supported v0.3 gates:

```text
discussion_gate
plan_doctor
draft_doctor
queue_doctor
promotion_preview
profile_binding_drift
```

v0.4 gates can already exist in code or schema, but they do not count as v0.3 closure unless explicitly tied to v0.3
handoff evidence:

```text
approval_gate
live_mapping_gate
cutover_readiness_gate
```

Gate result statuses must remain explicit:

```text
pass
warn
fail
blocked
needs_approval
error
```

Required behavior:

- Missing required workflow items -> `blocked`.
- Placeholder-only content -> `fail`.
- Non-placeholder ready marker / authored content can allow the relevant gate to `pass`.
- `promotion_preview` gate pass only proves mapping preview inputs are structurally ready.
- Gate pass cannot update downstream workflow item status by itself.
- Gate pass cannot create approval, execution package, run, lease or worker attempt.

## Transition Preview Requirements

v0.3 transition preview covers:

```text
from: promotion_preview
to: approval
required_gate: promotion_preview
```

It is read-only preview state:

```text
status = blocked when required gate is missing or not pass
status = ready when latest required gate result is pass
warnings include read-only / no-apply semantics
```

Transition preview does not promote queue candidates, does not write live mapping, does not create execution files and
does not modify managed projects.

## Approval Record Requirements

v0.3 approval record is an audit fact, not an execution permission.

An approval record must store:

```text
approval_kind
decision
scope_type
scope_id
actor
reason
risk_level
transition_preview_id
metadata
created_at
```

Required behavior:

- `approved` requires a ready transition preview.
- A blocked transition preview can only be rejected or left unresolved.
- `approval_is_execution=false` must remain explicit in metadata / evidence.
- Approval record can support later v0.4 gates, but it does not write AreaMatrix and does not start execution.
- Approval record must write command request, event and audit event.

## Profile Binding Drift

Every authored workflow version freezes the workflow profile binding at creation time. The `profile_binding_drift`
gate compares frozen binding to the current profile registry.

Required behavior:

```text
matching binding -> pass
missing frozen binding -> blocked
hash/version drift -> warn or blocked according to migration policy
profile migration attempted -> false in v0.3
```

Profile migration is a later explicit migration action. v0.3 must not silently mutate existing version gate rules when
the profile file changes.

## Allowed Reads

v0.3 may read:

```text
AreaFlow PostgreSQL metadata
AreaFlow workflow profile files under workflow/profiles/**
AreaFlow artifact store metadata
AreaMatrix project config metadata already registered in AreaFlow
```

v0.3 does not need to read AreaMatrix `workflow/versions/**` to create an authored version. Import / shadow paths may
still read managed project metadata under the v0.1/v0.2 contracts.

## Allowed Writes

v0.3 may write:

```text
AreaFlow PostgreSQL
AreaFlow-owned local artifact store
AreaFlow events
AreaFlow audit_events
AreaFlow command_requests
```

v0.3 must not write:

```text
AreaMatrix workflow/versions/**
AreaMatrix workflow/README.md
AreaMatrix .areaflow/status.json unless a separate status projection apply is authorized
AreaMatrix execution/progress/logs/checkpoints/run summaries
AreaMatrix source code
```

## Forbidden Actions

v0.3 must block:

```text
promotion apply
authoring cutover apply
execution cutover
write workflow/versions/**
write workflow/versions/*/execution/**
write progress.json
run ./task-loop run
start worker
create worker lease / heartbeat / attempt
call AI engine
resolve secret
run managed project shell command
run git mutation
publish release or create package
restore apply
```

## Handoff To v0.4

v0.3 handoff to v0.4 requires evidence that:

- An authored workflow version exists with frozen profile binding.
- Stage skeleton items and links exist and are project-scoped.
- AreaFlow-owned artifact placeholders or ready markers exist in the artifact store.
- Gate results are append-only and preserve blocked/fail/pass semantics.
- Transition preview is blocked or ready with explicit required gate relation.
- Approved approval record only exists after ready transition preview.
- Approval record remains `approval_is_execution=false`.
- Command requests, events and audit events exist for create/gate/transition/approval actions.
- Managed project protected paths remain unchanged.

v0.4 can then decide whether authoring cutover is allowed. v0.3 itself does not perform cutover.

## Anti-patterns

v0.3 must reject these interpretations:

- “A workflow version was created, so AreaMatrix authoring has cut over.”
- “Stage skeleton exists, so discussion / plan / draft content is complete.”
- “Placeholder artifacts exist, so gates can pass.”
- “promotion_preview gate passed, so promotion apply happened.”
- “transition preview ready, so approval is automatic.”
- “approval approved, so execution may start.”
- “profile binding drift passed once, so future profile changes can silently mutate existing versions.”
- “AreaFlow local artifact write means managed project files were written.”
