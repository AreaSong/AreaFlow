# v0.9 Desktop Shell Contract

## Purpose

本文定义 AreaFlow v0.9 `Desktop Shell` 的阶段边界。它补充
[`api-surface.md`](./api-surface.md)、
[`operations-deployment-observability-boundary.md`](../../../architecture/operations-deployment-observability-boundary.md)、
[`team-remote-control-boundary.md`](../../../../proposals/team-and-remote-control.md)、
[`auth-team-secret-boundary.md`](../../../../proposals/auth-team-secret.md) 和
[`../history/v1.0/milestones/v0.9-desktop-shell.md`](../milestones/v0.9-desktop-shell.md)。

v0.9 的目标是把 CLI/Web/API 已有的本机服务状态、dashboard launcher、worker pool 观察面和只读
desktop gates 放进 Tauri shell。v0.9 不打开真实 process supervision、OS notification bridge、native
tray/menu、secret resolve、team console、remote ops、workflow execution 或被管理项目写入。

核心规则：Desktop 是 AreaFlow API client，不是第二个 AreaFlow。Desktop 只能展示 API 返回的状态、
launcher 和 gate；不能直接维护 service、notification、tray、workflow、worker、secret、project 或
team 的第二状态源。

## Position In Roadmap

v0.9 位于 Multi-project Worker Pool 之后、Stable Platform 之前：

```text
Multi-project Worker Pool
-> Desktop Shell
-> Stable Platform
```

v0.9 只允许推进：

```text
Tauri shell scaffold
local service status display
dashboard launcher
service-control gate display
notification gate display
tray/menu gate display
selected project readiness display
worker pool observation
release/readiness preview display
local-only desktop configuration for API base and selected project
```

v0.9 不允许推进：

```text
real process start / stop / restart
background service supervisor
OS notification permission request
native tray/menu creation
settings that store plaintext secrets
secret resolve or secret injection
workflow execution from Desktop
worker scheduling from Desktop
AreaMatrix shim editing
execution cutover apply
execution forwarding apply / rollback
task-loop forwarding
release package / approval / rollout / publish apply
team console permission enforcement
remote ops control
managed upgrade or rollback apply
second local database
direct project file reads as business state
```

## Query Surface

v0.9 Desktop surfaces are read-only:

```text
GET /api/v1/service/status
GET /api/v1/desktop/service-control-gate
GET /api/v1/desktop/notification-gate
GET /api/v1/desktop/tray-menu-gate
GET /api/v1/projects/{project_key}/shim-authorization
GET /api/v1/projects/{project_key}/shim-apply-packet
GET /api/v1/projects/{project_key}/shim-apply-gate
GET /api/v1/projects/{project_key}/execution-cutover-readiness
GET /api/v1/projects/{project_key}/execution-forwarding-v1-readiness
GET /api/v1/projects/{project_key}/execution-forwarding-v1-apply-preview
GET /api/v1/projects/{project_key}/execution-forwarding-v1-apply-packet
GET /api/v1/projects/{project_key}/execution-forwarding-v1-command-preview
GET /api/v1/projects/{project_key}/execution-forwarding-v1-rollback-preview
GET /api/v1/release/rollout-plan-preview
areaflow service status --json
areaflow desktop service-control-gate --json
areaflow desktop notification-gate --json
areaflow desktop tray-menu-gate --json
```

Desktop may also display existing read-only Web/API surfaces such as project summary, worker pool preview, release
preview and audit records. These calls must stay under `/api/v1` and must not write local files, project files,
PostgreSQL state or artifact content.

## Service Status Boundary

`GET /api/v1/service/status` may aggregate:

```text
api health
database health
worker pool summary totals
dashboard URL
capabilities
forbidden_actions
generated_at
```

It must not:

```text
start service
stop service
restart service
open process supervisor
write desktop state into PostgreSQL
write managed project files
resolve secrets
schedule workers
create command requests
```

`service.status=ok` only means the observed API/service state is healthy enough to display. It does not mean Desktop
is authorized to manage the process lifecycle.

## Service Control Gate Boundary

`GET /api/v1/desktop/service-control-gate` may return:

```text
open_dashboard ready/enabled
start_service disabled/blocked
stop_service disabled/blocked
restart_service disabled/blocked
enable_notifications disabled/blocked
tray_menu disabled/blocked
required_capabilities
required_evidence
blockers
safety facts
```

Safety facts must keep:

```text
process_control_attempted=false
command_created=false
approval_created=false
audit_event_written=false
worker_scheduled=false
workflow_execution_started=false
project_write_attempted=false
secrets_resolved=false
network_used=false
```

The gate is an explanation layer. It must not create a command request, start/stop/restart a process, schedule a
worker, write audit events or mutate Desktop state. Real process supervision belongs to the operations ladder in
[`operations-deployment-observability-boundary.md`](../../../architecture/operations-deployment-observability-boundary.md).

## Notification Gate Boundary

`GET /api/v1/desktop/notification-gate` may explain:

```text
observe_event_stream read-only available
enable_system_notifications disabled/blocked
approval_needed_notifications disabled/blocked
run_failure_notifications disabled/blocked
worker_recovery_notifications disabled/blocked
required_previews
required_approvals
required_audit_events
blockers
safety facts
```

v0.9 must not request OS notification permission, open a persistent notification bridge, send remote notifications,
create command requests, schedule workers or store notification state as a second state source.

`observe_event_stream` means the UI can describe the requirement or reuse existing API/SSE observation where already
opened by Web. It does not mean Desktop has opened a native notification pipeline.

## Tray Menu Gate Boundary

`GET /api/v1/desktop/tray-menu-gate` may explain:

```text
open_dashboard launcher/read-only available
show_service_status read-only available
show_recent_events read-only available
start_service disabled/blocked
stop_service disabled/blocked
enable_notifications disabled/blocked
open_settings disabled/blocked
required_capabilities
blockers
safety facts
```

v0.9 must not create native tray/menu integration, request OS integration, execute service control, request notification
permission, open plaintext secret settings or schedule workers. A rendered panel in Tauri is not native tray/menu
creation.

## Desktop State Boundary

Desktop may keep local UI preferences such as:

```text
api_base_url
selected_project_key
last_refresh_timestamp
panel expansion state
```

These preferences are UI state only. They must not become workflow state, service state, notification state,
authorization state, worker state, artifact state or project source facts.

Desktop must not use local storage, Tauri storage, files or SQLite as a second AreaFlow database. If a future Desktop
cache is introduced, it must be explicitly cache-only, invalidatable and subordinate to `/api/v1`.

## Secret And Settings Boundary

Desktop may display secret readiness and secret reference names returned by API. v0.9 must not:

```text
read OS keychain
read env vars
read encrypted DB secret plaintext
store plaintext secrets in settings
inject secrets into workers or engines
test provider credentials
```

Secret resolve belongs to the R4 ladder in [`auth-team-secret-boundary.md`](../../../../proposals/auth-team-secret.md). Desktop
settings may later become a UI for secret references, but it still cannot bypass API permission, redaction and audit.

## AreaMatrix Boundary

Desktop may display selected AreaMatrix project status and blocked gates:

```text
shim authorization readiness
shim apply packet/gate review
execution cutover readiness
release rollout preview
worker pool preview
artifact metadata
audit records
```

It must not:

```text
edit AreaMatrix shim files
write workflow/README.md
write .areaflow/status.json
write workflow/versions/**
forward ./task-loop run
open shim apply command
open execution cutover apply
create release package
publish release
```

AreaMatrix remains an external managed project. Desktop presence does not change the AreaMatrix protected-path rules.

## Team And Remote Ops Boundary

v0.9 is local single-user shell. It must not become:

```text
team admin console
remote read-only console
remote command console
API token issuer
remote worker credential issuer
managed upgrade controller
remote rollback controller
support bundle exporter
telemetry uploader
```

Those capabilities follow [`team-remote-control-boundary.md`](../../../../proposals/team-and-remote-control.md) and
[`operations-deployment-observability-boundary.md`](../../../architecture/operations-deployment-observability-boundary.md).

## Handoff To v1.0

v0.9 handoff to v1.0 requires evidence that:

- Desktop only calls `/api/v1` read surfaces.
- Service status is read-only and does not create commands.
- Service-control gate keeps process control disabled/blocked.
- Notification gate does not request OS notification permission.
- Tray/menu gate does not create native tray/menu.
- Desktop gate CLI surfaces return the same read-only gate semantics as the API/Desktop shell.
- Desktop does not maintain second business state.
- Desktop displays selected project readiness without editing AreaMatrix.
- Desktop displays release/readiness preview without apply actions.
- Desktop build passes.
- AreaMatrix protected paths remain untouched.

v1.0 can then use Desktop as one observation surface for release, support and completion readiness. v1.0 must not treat
v0.9 Desktop panels as process supervision, release publish, support export or remote ops approval.

## Smoke Evidence

v0.9 evidence is split by boundary:

```text
service status:
  go test ./internal/project -run TestBuildServiceStatus -count=1
  go test ./internal/api -run TestServiceStatusEndpoint -count=1
  go run ./cmd/areaflow service status --json

desktop gates:
  go test ./internal/project -run 'TestBuildDesktop(ServiceControl|Notification|TrayMenu)Gate' -count=1
  go test ./internal/api -run 'TestDesktop(ServiceControl|Notification|TrayMenu)Gate' -count=1
  go test ./internal/app -run 'TestDesktop(ServiceControl|Notification|TrayMenu)GateToJSON' -count=1
  areaflow desktop service-control-gate --json
  areaflow desktop notification-gate --json
  areaflow desktop tray-menu-gate --json

desktop shell:
  cd desktop && npm run build

platform guard:
  go test ./...
  go build ./cmd/areaflow
```

If native Tauri packaging or OS integration is unavailable, v0.9 may still pass as Desktop shell scaffold when WebView
build and API-backed gate display evidence pass. That does not prove native tray, notification or process control.

## Anti-patterns

v0.9 must reject these interpretations:

- “Desktop shows service status, so it can restart the service.”
- “A notification gate panel exists, so OS notifications are enabled.”
- “A tray/menu gate panel exists, so native tray has been created.”
- “Desktop selected a project, so it can edit AreaMatrix files.”
- “Desktop stores API base/project key, so it owns platform state.”
- “Desktop shows execution cutover readiness, so it can forward task-loop.”
- “Desktop shows release rollout preview, so it can package or publish.”
- “Desktop shell exists, so team console or remote ops is open.”
