# v0.4 Workflow Ownership Cutover Contract

## Purpose

本文定义 AreaFlow v0.4 `Workflow Ownership Cutover` 的阶段边界。它补充
[`v0.3-version-authoring-contract.md`](./v0.3-version-authoring-contract.md)、
[`workflow-engine-contract.md`](../contracts/workflow-engine-contract.md)、
[`command-approval-contract.md`](./command-approval-contract.md)、
[`project-config.md`](./project-config.md)、
[`api-surface.md`](./api-surface.md)、
[`../migration/cutover-rollback-compat.md`](../migrations/cutover-rollback-compat.md)、
[`../migration/areamatrix-compatibility-shim-plan.md`](../migrations/areamatrix-compatibility-shim-plan.md) 和
[`../migration/areamatrix-workflow-migration.md`](../migrations/areamatrix-workflow-migration.md)。

v0.4 的目标是把新 workflow version 的 authoring source of truth 从项目目录候选态切换到 AreaFlow
PostgreSQL + AreaFlow artifact store。它不切 execution，不替代 `./task-loop run`，不写 AreaMatrix
`workflow/versions/**`，也不落地 AreaMatrix shim 文件，除非用户另行明确授权跨仓库编辑。

核心规则：v0.4 cutover 是 authoring cutover，不是 execution cutover。`cutover_readiness_gate=pass` 和
`project.cutover.apply status=applied` 只证明 AreaFlow DB 内的新 workflow version authoring ownership 已切换；
它们不能被解释为 task-loop forwarding、worker execution、source write、checkpoint、repair、engine、secret、
restore 或 publish 已打开。

## Position In Migration

v0.4 位于 AreaFlow authoring 建模之后、execution beta 之前：

```text
Shadow Doctor + Drift Check
-> New Version Authoring
-> Workflow Ownership Cutover
-> Runner Preview
-> Worker Execution Beta
```

v0.4 只允许推进：

```text
new workflow version authoring source of truth -> AreaFlow
legacy workflow -> read-only / immutable interpretation
compatibility commands -> read-only forward / fallback / blocked contract
```

v0.4 不允许推进：

```text
execution source of truth -> AreaFlow
./task-loop run forwarding
AreaMatrix workflow directory export
AreaMatrix shim file edits without explicit approval
```

## Minimal Closed Loop

v0.4 最小闭环是：

```text
workflow version create
-> mark required authoring items ready or provide non-placeholder content
-> promotion_preview gate
-> transition preview
-> approval record
-> approval_gate
-> live_mapping_gate
-> project compatibility
-> project shim-preview
-> project shim-readiness
-> project cutover-readiness
-> cutover_readiness_gate
-> project cutover-apply
-> protected path proof
```

The closed loop may be proven on a fixture project. Real AreaMatrix authoring cutover remains unclaimed until a
separate real dogfood evidence chain explicitly proves it.

## Slices

v0.4 has four slices:

```text
v0.4a:
  profile_binding_drift, approval_gate and live_mapping_gate.

v0.4b:
  compatibility contract, shim preview, shim readiness and shim authorization packet.

v0.4c:
  cutover readiness bundle and cutover_readiness_gate.

v0.4d:
  protected project.cutover.apply for AreaFlow DB-only authoring cutover.
```

v0.4d is still not AreaMatrix file cutover. It changes AreaFlow state only.

## Gate Requirements

v0.4 gates:

```text
profile_binding_drift:
  Compare frozen profile binding with current profile registry.

approval_gate:
  Verify latest approval record is approved and linked to a ready transition preview.

live_mapping_gate:
  Verify promotion_preview gate pass -> transition preview ready -> approval record approved.

cutover_readiness_gate:
  Persist the cutover readiness bundle as a gate result.
```

Required behavior:

- `profile_binding_drift` must not migrate profile rules.
- `approval_gate` must not create approval.
- `live_mapping_gate` must not write live execution mapping.
- `cutover_readiness_gate` must not run `project.cutover.apply`.
- Every gate must preserve `execution_write_attempted=false` where applicable.

## Compatibility Contract

`project compatibility` returns the command contract that an AreaMatrix shim can follow.

Allowed command modes:

```text
forward:
  Query or explicitly allowed AreaFlow command forwarding.

fallback_status:
  Read `.areaflow/status.json` for offline rough status.

blocked:
  Fail closed with a clear reason.
```

First shim command set:

```text
./dev workflow status
./dev workflow doctor
./dev workflow init --version <version>
./dev workflow open
./task-loop status
./task-loop run
```

Required command behavior:

- `./task-loop run` must be `blocked`.
- `./dev workflow doctor` must not implicitly pass `--allow-native`.
- `./dev workflow init --version <version>` can create AreaFlow-authored records only through AreaFlow Command API / CLI.
- No shim command may write AreaMatrix `workflow/versions/**`.
- Offline fallback can read `.areaflow/status.json`, but cannot treat it as the source of truth.

## Shim Preview / Readiness / Authorization

`shim-preview` is a read-only plan. It can list intended AreaMatrix file changes, forbidden paths, command mappings,
discovery order, verification commands and rollback steps. It must not perform those edits.

`shim-readiness` must remain blocked until all required evidence and explicit edit approval exist. Current required
items include:

```text
compatibility_contract
shim_preview
status_projection
task_loop_run_blocked
forbidden_paths_declared
rollback_plan_declared
real_areamatrix_readonly_smoke
real_areamatrix_status_projection_schema
areamatrix_dirty_worktree_review
explicit_edit_approval
```

`shim-authorization` is a read-only authorization packet. It must report:

```text
status = blocked
mode = read_only_authorization_packet
allowed files
forbidden paths/actions
required preflight
post-edit verification
rollback scope
safety_facts
next_required_approval
```

Producing the packet does not authorize edits. Real AreaMatrix shim implementation requires a later explicit user
approval that names allowed files, forbidden files/actions, validation and rollback.

`shim-apply-packet` and `shim-apply-gate` are the next read-only review layer. They turn one future shim edit
request into a machine-reviewable packet and go/no-go gate, but they still do not execute edits. The gate must require:

```text
allowed_files exactly match shim-authorization
authorization_snapshot_hash matches current authorization packet
expected_authorization_mode = read_only_authorization_packet
readiness blockers are limited to explicit_edit_approval
status_projection_packet_id
status_projection_gate_id
read_only_smoke_evidence_id
dirty_worktree_review_id
protected_path_fingerprint_id
rollback_plan_id
failure_mode = fail_closed
explicit approval id/actor/reason
idempotency_key
audit_correlation_id
```

这些 proof id 必须是 project-scoped proof reference，格式为
`<project_key>:<evidence_kind>:<id>`，其中 evidence kind 对应 status projection packet/gate、real read-only
smoke、dirty worktree review、protected path fingerprint 或 rollback plan。裸 ID 不能让 gate 通过。

If real AreaMatrix `.areaflow/status.json` is still legacy schema or any readiness evidence is missing, the shim
apply gate must remain `blocked` even when approval fields are present. `apply_command_eligible=true` only means the
packet can be submitted to a later protected command; it is not AreaMatrix edit authorization by itself.

## Cutover Readiness Bundle

`project cutover-readiness` aggregates:

```text
verification_bundle
status_mirror
compatibility_contract
workflow_version_authored
approval_gate
live_mapping_gate
rollback_plan
```

`v0.4-cutover-readiness` phase gate:

```text
pass:
  All required items pass, except accepted compatibility warnings.

blocked:
  Any required item is missing, fail, warn without exception, or blocked.
```

Accepted warnings must be explicit. Unknown drift, missing approval, missing live mapping, missing status mirror, missing
verification bundle, missing authored version or missing rollback plan cannot be accepted silently.

## Cutover Apply

`project cutover-apply` is the only v0.4 apply path. It is a protected Command API action that can only apply
`mode=authoring_cutover`.

It requires:

```text
workflow version import_mode = authored
project cutover-readiness phase_gate = pass
latest cutover_readiness_gate = pass
idempotency key / request hash
actor
reason
audit event
```

When allowed, it may update only AreaFlow state:

```text
workflow_versions.lifecycle_status = authoring_cutover
workflow_versions.status_summary.authoring_cutover.applied = true
workflow_owner = areaflow
execution_owner = project
project_write_attempted = false
execution_write_attempted = false
events
audit_events
command_requests response
```

It must return:

```text
project_write_attempted=false
execution_write_attempted=false
area_matrix_write_attempted=false
```

Any `mode` other than `authoring_cutover`, including `execution_cutover`, must be blocked.

## Rollback Semantics

Rollback is append-only:

```text
pre-cutover blocked:
  Record blockers; stay in shadow / authoring.

cutover apply blocked:
  Record denied command/event/audit; do not change lifecycle status.

cutover apply failed:
  Record failure event/audit; preserve prior AreaFlow state.

post-cutover drift:
  Freeze new authoring, run recovery doctor, optionally soft rollback through a later explicit command.

soft rollback:
  Mark new authoring version back to project-owned or read-only frozen; preserve events/audit/artifacts.
```

Rollback must not delete AreaFlow events, audit events, command requests, artifacts, workflow versions or historical
AreaMatrix workflow content.

## Allowed Reads

v0.4 may read:

```text

AreaFlow events/audit/command responses
AreaFlow-owned artifact metadata
AreaMatrix read-only import/shadow metadata already covered by v0.1/v0.2 contracts
AreaMatrix git status for protected path proof
```

v0.4 shim readiness may reference AreaMatrix file paths, but it cannot edit them without explicit user approval.

## Allowed Writes

v0.4 may write:

```text
AreaFlow PostgreSQL
AreaFlow events
AreaFlow audit_events
AreaFlow command_requests
AreaFlow workflow version lifecycle/status_summary for authoring_cutover
AreaFlow-owned artifact store when a command explicitly produces an AreaFlow artifact
```

v0.4 must not write:

```text
AreaMatrix workflow/versions/**

AreaMatrix .areaflow/status.json unless a separate status projection apply is authorized
AreaMatrix execution/progress/logs/checkpoints/run summaries

```

## Handoff To v0.5

v0.4 handoff to v0.5 requires evidence that:

- v0.3 authored workflow version exists and is project-scoped.
- promotion preview, transition preview, approval, approval gate and live mapping gate are recorded.
- compatibility contract is queryable and keeps `./task-loop run` blocked.
- shim readiness / authorization are read-only and blocked without explicit edit approval.
- cutover readiness bundle and `cutover_readiness_gate` can pass on fixture evidence.
- `project.cutover.apply` is idempotent and DB-only.
- `project.cutover.apply` blocks non-`authoring_cutover` modes.
- command response records no project write and no execution write.
- protected path proof is clean.

v0.5 can then introduce runner preview. It still must not treat authoring cutover as execution cutover.

## Anti-patterns

v0.4 must reject these interpretations:

- “cutover readiness passed, so cutover apply already happened.”
- “cutover apply happened, so execution cutover happened.”
- “AreaFlow owns authoring, so it may write AreaMatrix source files.”
- “shim authorization packet exists, so AreaMatrix shim edits are approved.”
- “`./task-loop run` is blocked in compatibility, but runner preview may forward it anyway.”
- “rollback means deleting AreaFlow audit or historical AreaMatrix files.”
- “workflow owner is AreaFlow, so execution owner is also AreaFlow.”
