# v0.1 Import Mirror Contract

## Purpose

本文定义 AreaFlow v0.1 `Import + Status Mirror` 的最小可运行边界。它补充
[`data-model-v0.1.md`](./data-model-v0.1.md)、[`project-config.md`](./project-config.md)、
[`areamatrix-import-scope-contract.md`](../../../architecture/areamatrix-import-scope-contract.md)、
[`api-surface.md`](./api-surface.md)、[`../dogfood/areamatrix-contract.md`](../../../dogfood/areamatrix-contract.md)
和 [`../migration/areamatrix-workflow-migration.md`](../migrations/areamatrix-workflow-migration.md)。

v0.1 的目标不是接管 workflow，而是让 AreaFlow 能从空 PostgreSQL 环境开始，注册 AreaMatrix，
只读导入 workflow metadata，并生成粗略 status projection。所有真实执行、cutover、AI engine、
worker scheduling、secret resolve 和 source write 都不属于 v0.1。

## Minimal Closed Loop

v0.1 关闭时，最小闭环是：

```text
migrate up
-> project add
-> project import
-> project import
-> project status / summary
-> project export-status or status-projection-apply
-> protected path proof
```

`project import` 必须可重复运行。第二次 import 可以创建新的 import run / snapshot，但不能修改
AreaMatrix 原文件，也不能把历史事实重写成新的完成状态。

## Required CLI Commands

v0.1 必须稳定支持以下 CLI：

```text
areaflow migrate up
areaflow migrate status
areaflow project add --config <path>
areaflow project list
areaflow project status <project_key>
areaflow project summary <project_key>
areaflow project import <project_key>
areaflow project import <project_key> --json
areaflow project status-projections <project_key>
areaflow project status-projection-authorization <project_key>
areaflow project status-projection-apply-packet <project_key>
areaflow project status-projection-apply-gate <project_key>
areaflow project status-projection-apply <project_key>
areaflow project export-status <project_key>
```

v0.1 `summary` 只要求提供 rough inventory、latest import 和 projection 状态。v0.2 才稳定
doctor / drift / stage coverage / native doctor 等 summary 字段。

`export-status` 是 legacy compatibility command，等价于对默认 `.areaflow/status.json` target 执行受保护
status projection apply。长期实现可以继续保留它，但主语义应收束到
`status-projection-apply-packet` -> `status-projection-apply-gate` -> `status-projection-apply`。
缺少完整 protected apply packet、apply gate 绑定字段或显式 approval 时，`export-status` 必须 fail
closed，记录 denied/blocked 审计事实，并保持 `project_write_attempted=false`；它不得绕过 packet/gate
直接写 `.areaflow/status.json`。

`status-projection-authorization` 是 apply 前的只读授权包预览。它必须列出 target、schema URI、
validator preflight、protected path check、write-set、expected-before preimage、rollback plan、permission
状态和 safety facts；它不得创建 command request、不得写 `.areaflow/status.json`、不得写
`workflow/README.md`，也不得把 `needs_approval` 解释为 apply 已打开。

`status-projection-apply-packet` 是 authorization preview 之后、apply gate 之前的只读 packet preview。
它从当前 authorization/preimage 自动生成 expected-before、source snapshot hash、schema URI、validator
preflight、protected path check、rollback action 和 accepted preimage schema status，并输出对应 CLI
apply command 和 API request。默认没有显式 approval 时状态必须是 `needs_approval`；带
`--explicit-approval` 也只生成可审查 packet，不执行 apply。
当目标是真实 AreaMatrix 的 `.areaflow/status.json` 时，authorization preview、packet preview、packet、API request 和 gate
必须暴露同一个机器可读 `required_authorization_phrase`，其值为
`授权执行 Package A，只允许写 AreaMatrix .areaflow/status.json`；fixture root 不要求该精确短语。

`status-projection-apply-gate` 是 packet preview 之后、真实 apply command 之前的只读 go/no-go 门禁。它消费
显式 packet 字段：target、expected-before preimage、source snapshot hash、schema URI、validator preflight、
protected path check、protected path fingerprint、rollback action、accepted preimage schema status 和 approval
actor/reason。
`apply_command_eligible=true` 只表示该 packet 可以进入受保护 `project.status_projection.apply` command；
packet preview 和 gate 自身不得创建 command request、不得写 status projection row、不得写项目文件。

`status-projection-apply` 必须携带与 apply gate 一致的 packet。缺少 source hash、expected-before、
schema preflight、protected path check、rollback action 或 explicit approval 时，command 必须写入
blocked/denied 审计事实并保持 `project_write_attempted=false`，不得调用 writer。

真实文件写入必须先做底层 root containment 校验：target 必须是项目根目录内的相对路径，拒绝空路径、
绝对路径、`../` 逃逸，以及通过 symlink parent 解析到项目根目录外的路径。通过路径校验后，再使用
同目录临时文件和 atomic replace：先写入、同步并关闭临时文件，再 rename 到 `.areaflow/status.json`。
成功后不得留下 `.status.json.tmp-*` 临时文件；失败时必须清理临时文件并保留原 preimage，随后按
rollback plan 复核 schema validator 和 protected path check。rollback compensation 只能在同目录短暂创建
`.status.json.rollback-*`，完成后不得留下 rollback 临时文件，且该临时路径不等于额外持久写入授权。

writer 在落盘前必须执行内建 stable projection 校验：`schema_version` 固定为 1，所有 schema required
string 字段必须非空，`rough_progress.percent` 必须在 0-100，`compatibility.blocked_commands` 必须非空，
生成的 JSON 只能包含 schema 允许的 top-level、`active_versions[]`、`rough_progress` 和 `compatibility`
字段，不得重新带入 `summary`、`generated_at`、`source` 或 `source_hash` 等 legacy 字段。

`status-projection-apply` 在调用 writer 前必须捕获 target preimage。若 writer 已成功，但后续
pre-commit DB 记录步骤失败，且当前文件 hash 仍等于本次 write hash，必须用 atomic restore 恢复
preimage；若 target 是本次新建文件则删除。若当前文件 hash 已变化，不得覆盖外部变化，必须 fail closed
并报告 rollback 失败。

以下命令可以已经存在，但不计入 v0.1 最小关闭条件：

```text
areaflow project doctor <project_key>
areaflow project readiness <project_key>
areaflow project import-diff <project_key>
areaflow project verify-bundle <project_key>
```

这些命令属于 v0.2 shadow / drift / readiness 证据面。它们不能被用来证明 v0.1 已经进入 shadow
阶段，也不能反过来要求 v0.1 具备 native doctor 或 stage equivalence。

## API Boundary

v0.1 API 必须支持 CLI/Web/Desktop 后续复用，但第一阶段可以由 CLI 直接调用 service layer。
稳定 API 路径仍以 `/api/v1` 为目标。

v0.1 Query API 最小面：

```text
GET /api/v1/health
GET /api/v1/projects
GET /api/v1/projects/{project_key}
GET /api/v1/projects/{project_key}/summary
GET /api/v1/projects/{project_key}/events?limit=20
GET /api/v1/projects/{project_key}/status-projections?limit=20
```

v0.1 Command API 最小面：

```text
POST /api/v1/projects
POST /api/v1/projects/{project_key}/import
POST /api/v1/projects/{project_key}/status-projections/apply
```

如果某个 v0.1 CLI 暂时直连 PostgreSQL/service layer，它的行为仍必须按上述 API 语义建模：
idempotency、actor、reason、permission、target URI、event 和 audit 不得被省略。

## Allowed Reads

AreaMatrix adapter v0.1 的 read envelope 和 minimum import set 以
[`areamatrix-import-scope-contract.md`](../../../architecture/areamatrix-import-scope-contract.md) 为准。简化口径是：v0.1 只允许读取
workflow/task index metadata，不复制历史原文。

AreaMatrix adapter v0.1 的读取上限包括：

```text
areaflow.yaml or example config
workflow/versions/**
workflow/residuals/**
workflow/README.md
progress.json metadata
logs/evidence/report path metadata
docs metadata needed for workflow linkage
git status only for protected path proof
```

读取大文件时应优先保存 metadata、hash、size、content type 和 source path；不把 prompt、日志、报告、
diff 或 evidence 原文塞入 PostgreSQL。

## Allowed Writes

v0.1 允许写入：

```text
AreaFlow PostgreSQL
AreaFlow-owned local artifact store when explicitly producing new AreaFlow artifact
AreaFlow audit/event rows
AreaMatrix/.areaflow/status.json only through status projection apply
```

写 `.areaflow/status.json` 必须同时满足：

- project config 中 `write_status=true`。
- target path 被 `write_paths` allowlist 覆盖。
- target path 不命中 `forbidden_paths`。
- command / apply 记录 actor、reason、idempotency key 和 safety facts。
- result 明确 `workflow_write_attempted=false`、`execution_write_attempted=false`、`engine_call_attempted=false`。

v0.1 不允许写 `workflow/README.md`。人读摘要的受控区块写入至少等到 authoring cutover 相关 gate 明确后
再打开。

## Forbidden Actions

v0.1 必须阻断：

```text
write workflow/versions/**
write workflow/versions/*/execution/**
write progress.json
write task-loop logs / runner lock / checkpoint / run summary
run ./task-loop run
run shell commands from managed project
create workflow version
promote queue
start worker
create lease / heartbeat / attempt
call AI engine
resolve secret
write source code or generated project files
run git mutation
publish release or create package
restore apply
```

`write_artifacts` 不等于写被管理项目文件。v0.1 即使允许 AreaFlow-owned artifact metadata，也不能把
AreaMatrix 历史 artifact 原文复制成 AreaFlow-owned content，除非后续 archive/copy gate 单独打开。

## Data Requirements

v0.1 import 至少要落入这些对象：

```text
projects
project_connections
project_permissions
workflow_versions
workflow_items
residuals
artifacts
project_status_snapshots
runs
events
audit_events
```

完整表层级、提前借用的 support tables 和 v0.1 inactive tables 以
[`data-model-v0.1.md`](./data-model-v0.1.md) 为准。特别是 `command_requests`、`project_configs`、
`status_projections` 和 `project_scheduling_policies` 即使在当前实现中可用，也只能为 v0.1 提供
idempotency、配置快照、projection history 和 metadata；不能被解释为通用 Command API、scheduler、
worker、engine 或 cutover 已打开。

`runs` 在 v0.1 只表示 import command run，不表示 execution run。status projection apply 主要由
`command_requests`、`status_projections`、`project_status_snapshots`、`events` 和 `audit_events` 表达。
任何 v0.1 `run` 记录都必须带有 mode / command class / safety facts，避免被后续 worker 误领取。

`areaflow.yaml` 字段解释、默认值、persistence mapping 和 v0.1 禁止项以
[`project-config.md`](./project-config.md) 为准。配置中出现 scheduling、engine profile、allowed commands
或 `enabled: true` 时，v0.1 仍只保存 metadata，不运行命令、不解析 secret、不创建 lease。

## Artifact Metadata Policy

v0.1 artifact metadata 必须区分：

```text
local:
  AreaFlow-owned artifact store 内容。

project_reference:
  仍留在 AreaMatrix 的历史文件、prompt、日志、evidence 或报告。

external_project:
  AreaFlow 只知道外部路径或 URI，不能声明可恢复。
```

`project_reference` 和 `external_project` 可以有 hash 和 source path，但不能因此被 report 成
restore-ready，也不能被 archive / GC 触碰。

AreaMatrix v0.1 当前实现把历史 project 文件保存为 `storage_backend=external_project`。导入候选文件、
artifact type mapping、hash / drift 语义和 non-import 路径见
[`areamatrix-import-scope-contract.md`](../../../architecture/areamatrix-import-scope-contract.md)。

## Status Projection Shape

`.areaflow/status.json` v0.1 只能保存粗略状态：

```text
project_id
project_name
area_flow_url
cutover_phase
active_versions[]
active_versions[].display_label
active_versions[].lifecycle_status
active_versions[].rough_progress.percent
active_versions[].rough_progress.label
active_versions[].rough_progress.blocked
last_synced_at
source_snapshot_hash
```

它不能保存完整 queue、execution attempt、logs、checkpoint、approval payload、secret、worker lease、
artifact 原文或 release evidence。

## Handoff To v0.2

v0.1 完成后，v0.2 才开始把 AreaFlow 与 AreaMatrix 原流程做 shadow 对照。进入 v0.2 前必须证明：

- AreaMatrix project 可重复注册或更新。
- import 可重复运行，且 diff 可由后续 v0.2 命令解释。
- status projection 只写允许路径。
- status projection authorization preview、apply packet preview 和 apply gate 在真实 AreaMatrix 上保持只读，
  且缺失 packet 时 fail closed。
- AreaMatrix protected path proof clean，或变化只限已授权 `.areaflow/status.json`。
- 没有 `workflow/versions/**`、execution、progress、logs、checkpoint 或 source code 变化。
- 没有 shell execution、task-loop run、engine call、secret resolve 或 worker lease。

## Smoke Evidence

v0.1 smoke 至少覆盖：

```bash
go test ./...
go build ./cmd/areaflow
AREAFLOW_DATABASE_URL=... ./scripts/smoke-fixture.sh
AREAFLOW_DATABASE_URL=... ./scripts/smoke-areamatrix-readonly.sh
git -C /Users/as/Ai-Project/project/AreaMatrix status --short -- \
  workflow/README.md \
  .areaflow/status.json \
  scripts/task_loop/console.py \
  scripts/dev_tools/cli.py \
  scripts/task_loop/runner.py \
  scripts/areaflow_shim.py \
  workflow/versions \
  workflow/versions/v1-mvp/execution/_shared/progress.json
```

`smoke-local.sh` 可以作为人工本机演练，但它默认 config 指向真实 AreaMatrix root，因此不能作为默认只读
证明。当 project root 解析为真实 AreaMatrix root 时，脚本必须先设置
`AREAFLOW_SMOKE_ALLOW_REAL_PROJECT_READ=1` 才会进入 `project add/import/summary/doctor`；随后它还会通过受保护的
status projection apply 流程写 configured project root 下的 `.areaflow/status.json`。脚本必须设置
`AREAFLOW_SMOKE_ALLOW_STATUS_APPLY=1` 才会进入 apply；当 project root 解析为真实 AreaMatrix root 时，还必须
额外设置 `AREAFLOW_SMOKE_ALLOW_REAL_PROJECT_STATUS_APPLY=1`。真实 AreaMatrix 只读证明应优先使用不会写
projection 的 smoke，或在单独授权后记录 projection apply evidence。

## Anti-patterns

v0.1 必须拒绝以下解释：

- “已经有 doctor，所以 v0.2 完成。”
- “`.areaflow/status.json` 写成功，所以 workflow 已 cutover。”
- “import 创建了 run，所以 worker execution 已打开。”
- “artifact hash 存在，所以历史 artifact 已归 AreaFlow 拥有。”
- “`write_artifacts=true`，所以可以写 AreaMatrix 文件。”
- “`project import` 运行了两次，所以 drift check 已完成。”
- “Web 能展示项目，所以 v0.7 已完成。”
