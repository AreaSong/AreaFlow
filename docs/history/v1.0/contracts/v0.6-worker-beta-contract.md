# v0.6 Worker Beta Contract

## Purpose

本文定义 AreaFlow v0.6 `Worker Execution Beta` 的阶段边界。它补充
[`v0.5-runner-preview-contract.md`](./v0.5-runner-preview-contract.md)、
[`execution-model.md`](../../../architecture/execution-model.md)、
[`execution-opening-strategy.md`](../../../../proposals/execution-opening.md)、
[`worker-scheduling-contract.md`](../../../architecture/worker-scheduling-contract.md)、
[`command-approval-contract.md`](./command-approval-contract.md)、
[`api-surface.md`](./api-surface.md) 和
[`../history/v1.0/milestones/v0.6-worker-beta.md`](../milestones/v0.6-worker-beta.md)。

v0.6 的目标是把 v0.5 的 dry-run run/task/attempt 模型推进到受控 worker lifecycle、lease、dry-run
worker loop 和有限的 scoped execution evidence。它不是 AreaMatrix execution cutover，不替代
`./task-loop run`，不打开真实 Codex CLI / engine execution，不解析 secret，不运行 shell，不写
AreaMatrix source / workflow execution / progress / logs / checkpoint。

核心规则：v0.6 的每一条能力只证明自己列出的 scope。fixture、read-only verify、artifact-only、
fixture/temp rollback drill、readiness 和 beta gate 不能互相累计成真实 AreaMatrix execution cutover。

## Position In Migration

v0.6 位于 runner preview 之后、Web dashboard 之前：

```text
Runner Preview
-> Worker Execution Beta
-> Web Dashboard
-> Multi-project Worker Pool
```

v0.6 只允许推进：

```text
worker registry / heartbeat
lease acquire / release / recover
dry-run worker run-once
engine adapter preview
execution approval gate
fixture-only execution apply
read-only verify
AreaFlow-owned artifact write
fixture project write rollback drill
fixture/temp generated write rollback drill
generated write readiness and beta approval gate
execution plan / project write / generated write gates
```

v0.6 不允许推进：

```text
AreaMatrix execution cutover
./task-loop run forwarding
real Codex CLI / AI engine call
secret resolution
network access from worker tasks
source write beta
retained real AreaMatrix generated apply
checkpoint apply
repair apply
publish / restore apply
```

## Scope Ledger

v0.6 capabilities must be reported with one of these scope labels:

```text
preview_only:
  read-only preview/readiness/gate. No command request unless explicitly listed, no task claim, no artifact.

area_flow_state_only:
  writes AreaFlow PostgreSQL state and audit facts only.

artifact_store_only:
  writes AreaFlow-owned artifact store and PostgreSQL metadata, but not managed project files.

fixture_only:
  applies only to fixture/temp project roots and must leave rollback evidence.

read_only_project:
  reads allowlisted managed project files for hash/size evidence only, never stores original content.

generated_rollback_drill:
  fixture/temp generated-only write/verify/rollback; no retained real AreaMatrix result.

closed:
  not opened in v0.6.
```

Any CLI, API, Web, Desktop or evidence page that surfaces a v0.6 result must preserve the scope label or equivalent
safety facts. A `pass` in one scope is not a `pass` in another scope.

```text
areaflow run execution-gate <run_id>

areaflow run project-write-design-gate <run_id>
areaflow run fixture-queue <project_key> <version_label>

```

Every write-like command must use Command API semantics: idempotency key/request hash, actor/reason, permission,
gate/approval where applicable, event and audit evidence.

## API Boundary

v0.6 Query API may expose worker list, engine preview, execution gates, execution plan, generated write readiness and
beta gates. Query API must not create command requests, leases, attempts, artifacts or audit rows.

v0.6 Command API may create AreaFlow worker lifecycle rows, leases, scoped runs/tasks/attempts/artifacts, events,
audit events and command responses. Command API must not write managed project files except inside explicitly
fixture/temp rollback drill scopes.

## Worker Lifecycle Boundary

Worker register and heartbeat are `area_flow_state_only`.

They may write:

```text
workers
worker_heartbeats
actors
events
audit_events
command_requests
```

They must not create:

```text
leases
run_attempts
artifacts
worker_run_once attempts
project writes
engine calls
commands
secrets
network
```

Required response facts:

```text
project_write_attempted=false
execution_write_attempted=false
engine_call_attempted=false
commands_run=false
secrets_resolved=false
network_used=false
lease_created=false
attempt_created=false
artifact_created=false
worker_run_once=false
```

## Lease Boundary

Lease acquire/release/recover are `area_flow_state_only`.

They may create or update `leases`, change eligible `run_task.status` to lifecycle states such as leased,
released or needs_recovery, and write event/audit/command facts. They must not execute the task.

Capability denial must:

```text
decision=denied
lease_created=false
attempt_created=false
artifact_created=false
worker_run_once=false
```

Lease recovery may mark expired active leases as `needs_recovery`; it must not mark tasks failed, passed or done.

## Dry-run Worker Run-once Boundary

Worker `run-once` for dry-run tasks is `artifact_store_only` when it claims a task and `area_flow_state_only` when
idle. It may claim an existing dry-run `run_task`, create one dry-run `worker_run_once` attempt, write a
`worker_run_once_report` artifact and release the lease.

It must not:

```text
perform copy / verify
call engine
run shell
read or write managed project files
resolve secrets
access network
turn dry-run task evidence into real execution evidence
```

## Engine Adapter Preview Boundary

Codex CLI adapter preview is `preview_only`.

It may read project config, permission rows and engine profile metadata. It may return command preview,
capability/path preflight and artifact redaction plan.

It must keep:

```text
engine_call_attempted=false
commands_run=false
secrets_resolved=false
network_used=false
project_write_attempted=false
execution_write_attempted=false
artifact_created=false
```

Even if all preview checks pass, real Codex CLI execution remains closed until later no-secret engine execution
approval and budget/redaction policies are in place.

## Execution Approval Gate Boundary

Execution approval gate is `preview_only`. It may read a run, its task candidates, approval records, gate results,
engine preview and workers.

It must block:

```text
dry-run runner preview run
missing approval
missing approval gate
missing live mapping gate
missing worker capability
closed engine / command / secret / network capability
```

It must not create command requests, leases, attempts, artifacts or worker starts.

## Scoped Execution Boundaries

### Fixture Execution

`fixture execution` is `artifact_store_only` and proves only AreaFlow PG + artifact store execution apply. It may
create a lease, attempt, `fixture_execution_report`, task passed state and run passed state. It must not read or write
managed project files, run shell or call engine.

### Read-only Verify

`read-only verify` is `read_only_project`. It may read only an allowlisted target file and store path/hash/size
evidence. It must not store file content, modify files, run commands, call engine or treat verify success as source
write completion.

### Approved Artifact Write

`approved artifact write` is `artifact_store_only`. It may write AreaFlow-owned artifact content and PG metadata.
It must not read or write managed project files.

### Fixture Project Write

`fixture project write` is `fixture_only`. It may modify only an existing ordinary file under a fixture/temp project
root, after expected-before hash/size checks. It must write preimage, copy, verify and rollback evidence, and must
restore the preimage before completion.

### Managed Generated Write

`managed generated write` in v0.6 is `generated_rollback_drill`. It is restricted to fixture/temp projects and
generated-only paths such as `.areaflow/generated/**` or `.areamatrix/generated/**`. It must restore the preimage
before completion and keep:

```text
real_areamatrix_write_opened=false
fixture_or_temp_project_only=true
rollback_verified=true
```

## Generated Write Readiness And Beta Gate

Generated write readiness and generated write apply beta gate are `preview_only`.

They may report whether config, permission rows, allowlisted generated paths, rollback contract and approval evidence
are ready for review. They must keep:

```text
apply_open=false
real_areamatrix_write_opened=false
project_write_attempted=false
task_claimed=false
lease_created=false
attempt_created=false
artifact_created=false
```

Even when `ready_for_review=true`, real AreaMatrix generated-only apply remains closed until a later explicit R3
approval and focused smoke evidence.

## Allowed Writes

v0.6 may write within its scoped boundaries:

```text
AreaFlow PostgreSQL
AreaFlow command_requests
AreaFlow workers / worker_heartbeats / leases
AreaFlow runs / run_tasks / run_attempts
AreaFlow artifact metadata
AreaFlow events / audit_events
AreaFlow-owned local artifact store
fixture/temp project files only in rollback drill scopes
```

v0.6 must not write:

```text
real AreaMatrix source files
real AreaMatrix workflow/versions/**
real AreaMatrix workflow/versions/**/execution/**
real AreaMatrix progress.json / logs / checkpoints
real AreaMatrix workflow/README.md
real AreaMatrix .areaflow/status.json from worker beta
managed project source files outside fixture/temp drill scopes
retained generated/projection output in real AreaMatrix
```

## Forbidden Actions

v0.6 must block:

```text
AreaMatrix execution cutover
./task-loop run forwarding
real Codex CLI execution
AI engine calls
shell command execution from worker tasks
secret resolution
network access from worker tasks
source write beta
retained real AreaMatrix generated apply
checkpoint apply
repair apply
git mutation / checkpoint
restore apply
publish apply
```

## Status Semantics

Legacy/scoped run_task statuses such as `verified`, `artifact_written` and `rollback_verified` may exist as
compatibility states in v0.6. They must be interpreted as scoped outcomes and eventually converge to:

```text
status=passed
outcome=read_only_verify_passed | artifact_write_passed | rollback_verified | generated_write_rollback_verified
```

New v0.6 capabilities must not introduce more ability-specific status enums when outcome, attempt metadata, artifact
or gate evidence is sufficient.

## Handoff To v0.7

v0.6 handoff to v0.7 requires evidence that:

- worker register/heartbeat are idempotent and do not create leases/attempts/artifacts.
- lease acquire/release/recover enforce active lease uniqueness and capability denial.
- worker run-once dry-run can create worker-run evidence without copy/verify/project writes.
- engine preview remains preview-only and blocked by disabled profile/capabilities where appropriate.
- execution approval gate blocks dry-run preview runs and missing approval/capability states.
- fixture execution, read-only verify and approved artifact write prove only their own scopes.
- fixture project write and managed generated write are restricted to fixture/temp rollback drills.
- generated write readiness/beta gate remain no-apply and real AreaMatrix apply closed.
- AreaMatrix protected path proof is clean.

v0.7 can then display these states through Web. Web must not reinterpret v0.6 scoped evidence as execution cutover.

## Anti-patterns

v0.6 must reject these interpretations:

- “worker registered, so execution is open.”
- “lease acquired, so task copy/verify happened.”
- “run-once passed, so real worker execution passed.”
- “fixture execution passed, so AreaMatrix execution cutover is ready.”
- “read-only verify passed, so source write is safe.”
- “approved artifact write passed, so project write is allowed.”
- “fixture/temp rollback drill passed, so real AreaMatrix generated apply can be retained.”
- “generated-write readiness is ready, so apply is open.”
- “execution approval gate pass can bypass Command API.”
- “scoped v0.6 evidence can be accumulated into `./task-loop run` forwarding.”
